Blocking Apps from Drawing
Under the Navigation Bar

Some apps used the Android window-inset API to render content behind the navigation bar, visually breaking the SystemUI layout. I traced the call chain into the framework, patched the setter in PhoneWindow.java, and enforced the correct behaviour system-wide as a temporary measure before a release milestone.

1. The Problem

Android provides an API that lets app developers declare that their window should draw behind system bars — including the navigation bar — so they can extend their content edge-to-edge. In Java/Kotlin this is exposed via:

Kotlin Any app's Activity.kt 
// Tells the framework: "let my window draw behind the nav bar"
WindowCompat.setDecorFitsSystemWindows(window, false)

// Older equivalent via legacy flags
window.addFlags(WindowManager.LayoutParams.FLAG_LAYOUT_NO_LIMITS)

On a normal consumer Android device this is intentional — apps handle insets manually and the OS does not care. On our automotive head-unit, however, the navigation bar is a SystemUI component that must always remain fully visible and unobstructed. Several third-party apps that were integrated into the vehicle had used this API (sometimes incorrectly, without then handling the insets themselves), causing their content to slide underneath the navigation bar and visually overlap SystemUI.

 Symptom

App content rendered behind the navigation bar, making nav bar buttons appear on top of app UI elements.

 Root Cause

Apps called setDecorFitsSystemWindows(false) or set FLAG_LAYOUT_NO_LIMITS without properly consuming window insets.

 Urgency

Discovered close to a release milestone — a clean, guaranteed fix was needed immediately, even if temporary.

2. Tracing the API into the Framework

To understand exactly what the API does under the hood — and where to intercept it — I traced the call chain from the public SDK surface down into the AOSP framework source.

1
App calls WindowCompat.setDecorFitsSystemWindows(window, false) AndroidX compatibility wrapper — delegates directly to window.setDecorFitsSystemWindows(false) on API 30+.
2
android.view.Window.setDecorFitsSystemWindows(boolean) Abstract method in the SDK-level Window class (android.view package). The concrete implementation lives in the framework.
3
PhoneWindow.setDecorFitsSystemWindows(boolean) The real implementation in com.android.internal.policy.PhoneWindow. This is where the flag is actually applied to the window's DecorView and its WindowManager.LayoutParams.
4
DecorView adjusts layout & dispatches insets The DecorView re-configures its layout fit flags and re-dispatches WindowInsets to child views, allowing app content to expand into system bar areas.

PhoneWindow.java lives at frameworks/base/core/java/com/android/internal/policy/PhoneWindow.java in the AOSP source tree. It is the single concrete implementation of android.view.Window on all Android phones and automotive targets. Every app window — activity, dialog, or system — goes through this class.

3. The Framework Patch

Once I identified PhoneWindow.setDecorFitsSystemWindows() as the entry point, the fix was straightforward: override the method to always enforce that the decor fits system windows, regardless of what the calling app requests. By returning early after forcing the value to true, the framework ignores any app's attempt to draw under the navigation bar.

Java frameworks/base/core/java/com/android/internal/policy/PhoneWindow.java 
// ORIGINAL — honours the caller's request
@Override
public void setDecorFitsSystemWindows(boolean decorFitsSystemWindows) {
    mDecorFitsSystemWindows = decorFitsSystemWindows;
    applyDecorFitsSystemWindows(decorFitsSystemWindows);
}

// PATCHED — always enforce true; app request is silently ignored
@Override
public void setDecorFitsSystemWindows(boolean decorFitsSystemWindows) {
    // [GVM-TEMP] Block apps from drawing under system bars before release milestone.
    // Any app passing false is overridden here. Remove once apps are fixed.
    mDecorFitsSystemWindows = true;
    applyDecorFitsSystemWindows(true);
}

This change was applied to the platform source and built into the system image — not an app-level workaround. It affected every window on the device, guaranteeing no app could bypass it regardless of which API variant they used.

Why this approach?

  • Single enforcement point — one change in the framework covers all apps, including those that use the legacy FLAG_LAYOUT_NO_LIMITS path funnelled through the same decor logic.
  • Zero risk of regression in SystemUI itself — SystemUI does not call setDecorFitsSystemWindows(false) on its own windows.
  • Immediately verifiable — after rebuilding and flashing the system image, the visual overlap was gone on all affected apps without touching any app APK.

4. Pre-release Context & Temporary Nature

The patch was introduced as a deliberate short-term measure ahead of a release milestone. The priority was to ship a build where the navigation bar was guaranteed to be unobstructed — negotiating individual fixes with each app team would have taken longer than the available window.

 Milestone gate

Navigation bar must be fully visible and unobstructed on all integrated apps before release sign-off.

 Time constraint

Insufficient time to coordinate fixes across multiple independent app teams before the deadline.

 Risk profile

Framework change was low-risk: it made the system more restrictive, not more permissive, and had no side-effects on SystemUI itself.

The patch included a clear comment (// [GVM-TEMP]) marking it as temporary so it would not be silently carried forward after the milestone.

5. Follow-up: Requesting App-side Fixes

After the milestone passed, I identified all affected apps by searching for calls to setDecorFitsSystemWindows(false) and FLAG_LAYOUT_NO_LIMITS across the integrated app source repositories. For each offending app I raised a ticket with the relevant team, documenting:

  • Which API call was incorrect and why it is inappropriate on the automotive target.
  • The correct approach — either remove the call entirely, or, if edge-to-edge content was genuinely required, properly consume WindowInsets in their view hierarchy to avoid overlapping the nav bar.
  • A deadline for the fix, after which the framework patch would be reverted.

Once all app teams had submitted corrected builds and the fixes were verified on-device, the temporary override in PhoneWindow.java was removed and the platform was rebuilt, restoring the standard Android behaviour for any app that legitimately needed it.
