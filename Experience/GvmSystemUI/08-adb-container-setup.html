1. System Architecture: Linux + Android Container

The GM vehicle platform (Qualcomm SA8155) runs Yocto Linux as the host OS. Android 15 runs as an LXC (Linux Container) guest named android1 inside that Linux OS. The two systems share the same kernel but have isolated user-spaces.

Host OS

Yocto Linux
Hardware abstraction, bootloader, LXC daemon, network, USB stack, adb endpoint

Guest

Android 15 (LXC container: android1)
SystemUI, apps, VHAL, CarService — all standard Android services running inside the container namespace

ADB traffic enters through the Linux USB stack, is forwarded by usb.service (or via the USB RC file) into the Android container's adbd daemon. Without configuration, this chain is broken by default — the USB init scripts, network bridges, and verity protections all must be handled before ADB works.

The LXC container's rootfs is mounted at runtime from the physical partitions (system, system_ext, vendor) on the flash storage. By default these are read-only dm-verity protected partitions — you cannot push files to them without disabling verity and remounting with OverlayFS.

2. The Problem

A fresh image flash leaves the device in a state where:

  • dm-verity is enabled — the kernel verifies every block of the system partitions against a cryptographic hash tree. Any attempt to write to /system, /system_ext, or /vendor causes a kernel panic or boot loop.
  • ADB is not routed to the container — the USB init RC file (init.qcom.usb.rc) that starts adbd inside the Android container needs to be configured correctly.
  • Network bridge for ADB-over-TCP is not set up — the container's network interfaces (reth0, reth1) need to be brought up.
  • System partitions are read-only — even after disabling verity, simply pushing an APK to /system_ext/priv-app/ fails because the partitions are not remounted as writable.

The solution is a sequence of two scripts — mount.bat and adb-setup.bat — plus supporting files, that together solve all four problems in the correct order.

3. Step 1 — Disable dm-verity (verity.bat)

This must be done once after every image flash, before mount.bat. It requires a reboot to take effect.

Batch setup/verity.bat 
@echo off
adb wait-for-device
echo "root & mount linux"
adb root
adb disable-verity    REM disables dm-verity, requires reboot to take effect

adb disable-verity patches the AVB (Android Verified Boot) metadata to mark the device as "unlocked-for-development". After the required reboot, the kernel will no longer validate partition block hashes, making them writable via remount or OverlayFS.

If you run mount.bat without disabling verity first, the overlay mount will succeed on the Linux host side but the Android container will fail to boot — the kernel detects modified data blocks and refuses to mount the partition.

4. Step 2 — mount.bat: OverlayFS + pre-mount Script

mount.bat configures the Android container to use OverlayFS for /system and /system_ext. This makes them writable at the overlay layer while the underlying partitions remain read-only — no flash erase required. Changes persist in /data/overlay/.

Batch setup/mount.bat 
@echo off
echo "WARNING: YOU NEED TO DISABLE VERITY FIRST"
adb wait-for-device
adb root
adb shell "setenforce 0"                       REM permissive SELinux — needed on host Linux
adb shell "mount -o rw,remount /"               REM make Linux rootfs writable

REM Create OverlayFS upper + workdir directories in /data (persists across reboots)
adb shell "mkdir /data/overlay"
adb shell "mkdir /data/overlay/system_ext"
adb shell "mkdir /data/overlay/system_ext/upper"
adb shell "mkdir /data/overlay/system_ext/workdir"
adb shell "mkdir /data/overlay/system"
adb shell "mkdir /data/overlay/system/upper"
adb shell "mkdir /data/overlay/system/workdir"

REM Push the pre-mount shell script into the LXC config directory on the device
REM LXC executes this script before starting the android1 container
adb push F:\01.env\05.gvm\setup\pre-mount.sh /etc/lxc/android1/pre-mount.sh

REM Push the LXC container config (network bridge, device mappings)
adb push F:\01.env\05.gvm\src\myconfig /etc/lxc/android1/config

adb shell "chmod 755 /etc/lxc/android1/pre-mount.sh"
adb shell "sync"

What OverlayFS achieves

OverlayFS merges two directory trees into one view:

LayerLocationAccess
lowerPhysical partition (/dev/mapper/system_ext)Read-only
upper/data/overlay/system_ext/upperRead-write (on eMMC /data)
work/data/overlay/system_ext/workdirInternal bookkeeping
merged (view)${LXC_ROOTFS_PATH}/system_extAndroid container sees this — reads come from lower, writes go to upper

The result: adb push GvmSystemUI.apk /system_ext/priv-app/GvmSystemUI/GvmSystemUI.apk writes to /data/overlay/system_ext/upper/priv-app/... — the physical partition is never touched, and the change survives reboots.

5. Inside pre-mount.sh: Partition Mounting Logic

This shell script is executed by the LXC daemon on the Linux host immediately before the Android container starts. It is the bridge between the Linux block devices and the Android rootfs. The script handles three cases based on what block devices are present:

Shell setup/pre-mount.sh (key sections) 
#!/bin/sh
LXC_OVERLAY_FILES_PATH=/etc/lxc/overlay/${LXC_NAME}

if [ -e "/dev/device-mapper" ]; then
    # Case 1: dm-verity device-mapper present → production image
    # Detect A/B boot slot and mount the correct slot's partitions
    boot_slot=$(abctl --boot_slot 2>/dev/null | cut -c 1-2)  # → "_a" or "_b"

    # Mount system partition (verity-mapped or raw device)
    if [ -e "/dev/mapper/system-verity" ]; then
        mount /dev/mapper/system-verity ${LXC_ROOTFS_PATH}
    else
        mount /dev/device-mapper/system${boot_slot} ${LXC_ROOTFS_PATH}
    fi

    # Same pattern for vendor, system_ext, system_dlkm, vendor_dlkm, product...
    mount /dev/mapper/system_ext-verity ${LXC_ROOTFS_PATH}/system_ext
    # ...etc

elif [ -h "/dev/disk/by-partlabel/la_system" ]; then
    # Case 2: non-verity GPT label layout (older/dev images)
    mount /dev/disk/by-partlabel/la_system ${LXC_ROOTFS_PATH}
    mount /dev/disk/by-partlabel/la_vendor ${LXC_ROOTFS_PATH}/vendor
fi

# Mount OverlayFS → system_ext becomes writable through /data/overlay
mount -t overlay -o \
    lowerdir=${LXC_ROOTFS_PATH}/system_ext,\
    upperdir=/data/overlay/system_ext/upper,\
    workdir=/data/overlay/system_ext/workdir,\
    userxattr,override_creds=off \
    overlay ${LXC_ROOTFS_PATH}/system_ext

mount -t overlay -o \
    lowerdir=${LXC_ROOTFS_PATH}/system,\
    upperdir=/data/overlay/system/upper,\
    workdir=/data/overlay/system/workdir,\
    userxattr,override_creds=off \
    overlay ${LXC_ROOTFS_PATH}/system

# Shared binds: camera socket, firmware, bluetooth, DSP, VHAL, factory
mount --bind -rw /dev/vhal  ${LXC_ROOTFS_PATH}/vendor/var/vhal
mount --bind   /factory     ${LXC_ROOTFS_PATH}/vendor/var/factory
mount --bind -r /firmware   ${LXC_ROOTFS_PATH}/vendor/firmware_mnt
mount --bind -r /bluetooth  ${LXC_ROOTFS_PATH}/vendor/bt_firmware
mount --bind -r /dsp        ${LXC_ROOTFS_PATH}/vendor/dsp
mount --bind -rw /tmp       ${LXC_ROOTFS_PATH}/vendor/var/socket/camera

The userxattr and override_creds=off overlay options are critical for Android: they allow the Android container (running as root inside its namespace) to set SELinux xattrs on files in the overlay upper layer, even though the overlay itself is mounted on the Linux host side without full Android credentials.

6. Step 3 — adb-setup.bat: USB & Network Init

After rebooting with the new pre-mount configuration, adb-setup.bat pushes the USB init RC file and network script that enable ADB routing into the Android container. It also disables the Linux host's competing usb.service.

Batch setup/adb-setup.bat 
adb wait-for-device
adb root
adb shell "setenforce 0"
adb shell "mount -o remount,rw /"

REM Push custom USB init RC into the container's vendor overlay
REM This configures the USB gadget mode so Android's adbd starts correctly
adb push F:\01.env\05.gvm\src\init.qcom.usb.rc \
         /etc/lxc/overlay/android1/vendor/etc/init/hw/
adb shell "chmod 644 /etc/lxc/overlay/android1/vendor/etc/init/hw/init.qcom.usb.rc"

REM Push the network init script (brings up reth0/reth1 bridge interfaces)
adb push F:\01.env\05.gvm\src\init.qti.container.network.sh \
         /etc/lxc/overlay/android1/vendor/bin/
adb shell "chmod 755 /etc/lxc/overlay/android1/vendor/bin/init.qti.container.network.sh"

REM Sync the Android container's filesystem
adb shell "lxc-attach android1 -- sync"

REM Mask the Linux host USB service so it doesn't compete with Android's adbd
adb shell "systemctl mask usb.service"
adb shell "sync"

What each push achieves

File pushedDestinationEffect
init.qcom.usb.rc /etc/lxc/overlay/android1/vendor/etc/init/hw/ Android init reads this RC file and configures the USB gadget driver to expose the ADB interface. Without it, the USB device presents no ADB function.
init.qti.container.network.sh /etc/lxc/overlay/android1/vendor/bin/ Brings up reth0/reth1 virtual Ethernet interfaces that bridge the Android container to the Linux host network — enabling ADB-over-TCP fallback and network access.

Files pushed to /etc/lxc/overlay/android1/vendor/... land in the vendor overlay — a separate OverlayFS layer above the raw vendor partition. This means the pushed files survive reboots and don't require reflashing but also don't touch the factory image.

7. Daily Developer Workflow

①
First time after image flash only: Disable verity Run verity.bat → reboot device.
②
First time after image flash only: Configure mounts Run mount.bat — sets up OverlayFS directories, pushes pre-mount.sh and LXC config → reboot device.
③
First time after image flash only: Enable ADB Run adb-setup.bat — pushes USB RC and network script → reboot.
④
Everyday: Install APK Build APK with Gradle → run install-gvm-systemui.bat which pushes APK and restarts SystemUI.
⑤
If ADB USB connection drops Run adb-interface.bat → invokes enable_adb.ps1 to re-enable the Windows USB PnP device.

Install script

Batch install/install-gvm-systemui.bat 
adb wait-for-device
REM Kill the running SystemUI process inside the Android container
adb shell "killall com.android.systemui"
REM Restart SystemUI on both displays (FPI + RSI)
adb shell "am start-service com.android.systemui/.SystemUIService"
adb shell "am start-service com.android.systemui/.SystemUIService --display 1"

PowerShell enable_adb.ps1 
# Re-enables Windows ADB USB device that got Error/Disabled by power management
$deviceName = "Android Composite ADB Interface"
$devices = Get-PnpDevice | Where-Object { $_.FriendlyName -like "*$deviceName*" }

foreach ($dev in $devices) {
    if ($dev.Status -eq "Error" -or $dev.Status -eq "Disabled") {
        Enable-PnpDevice -InstanceId $dev.InstanceId -Confirm:$false
    }
}

9. Why This Approach?

Design Rationale

The alternative to this setup is reflashing the entire system image for every code change — a process that takes 10–20 minutes on the Qualcomm SA8155 flasher. With OverlayFS + ADB, a code change can be pushed and tested in under 20 seconds: adb push + killall com.android.systemui. The OverlayFS approach is superior to a simple adb remount + direct write because it requires no changes to the raw flash partitions — the device stays in a known-good factory state, and all developer changes exist only in the /data/overlay layer, which can be wiped cleanly by deleting those directories. The pre-mount.sh mechanism is the standard LXC hook point on this platform — injecting OverlayFS into it means the writable mounts are active every time the container starts, even after a reboot, without any manual intervention.

No reflash needed

Push and test any APK or framework file in seconds. Device stays on factory image baseline.

Persistent changes

OverlayFS upper layer is on /data — survives reboots. No repeat setup after power cycle.

Clean rollback

Delete /data/overlay/system/upper → device reverts to factory state. Zero risk to base image.

Multi-display testing

fpi.bat / rsi.bat switch display resolution to test all 3 screen configurations without physical hardware swap.
