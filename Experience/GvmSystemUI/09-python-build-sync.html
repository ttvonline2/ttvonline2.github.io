1. The Problem

The AOSP platform build for the GM vehicle project runs on a remote Linux build server. The full build pipeline before this tooling existed looked like this:

1
Edit source locally Changes to GvmSystemUI on Windows machine (E:\01.Source\)
2
Manually copy to build server Open WinSCP / scp, navigate to the correct directory, drag files. ~3–5 min of manual work.
3
SSH in and trigger build Connect to server, cd to AOSP root, run build command. ~15–30 min wait.
4
Manually download APK Find the output APK in out/target/product/fpi/..., download via WinSCP.
5
Manually push to device adb push + adb shell killall + adb shell am start-service. Multiple commands.

Total: 20–40 minutes per iteration, with 5–10 minutes of manual steps repeated every cycle. The Python tool reduces this to one command.

4. Source Sync: git patch Approach

The most elegant part of the tool is how it syncs local changes to the server. Instead of uploading entire files (slow, risks overwrites), it uses git diff to create a patch file and applies it on the server side:

Python build.py — copy_source_to_server() 
def copy_source_to_server():
    # 1. On Windows: stage all modified/new files, generate unified diff patch
    os.system('cd /d "' + LOCAL_SOURCE_PATH + '" && git add . && git diff HEAD > changes.patch')
    changesPath = os.path.join(LOCAL_SOURCE_PATH, 'changes.patch')

    # 2. On the server: reset to HEAD (clean state), then apply the patch
    util.exec(server_ssh,
        f'cd {SERVER_SOURCE_PATH} && rm -rf changes.patch && git reset --hard && git clean -df')

    # 3. Send only the patch file (tiny — kilobytes) via SFTP
    with server_ftp.cd(SERVER_SOURCE_PATH):
        server_ftp.put(changesPath, callback=progress_cb)

    # 4. Apply patch on server → source tree matches local edits exactly
    util.exec(server_ssh, f'cd {SERVER_SOURCE_PATH} && git apply changes.patch')
    util.exec(server_ssh, f'cd {SERVER_SOURCE_PATH} && rm -rf changes.patch')
    util.exec(server_ssh, f'cd {SERVER_SOURCE_PATH} && git status')

    # 5. Clean up local patch file, unstage local git index
    util.remove_entry(changesPath)
    os.system('cd /d "' + LOCAL_SOURCE_PATH + '" && git reset')
Why git patch?

A typical code change modifies 3–10 files out of thousands. Uploading a patch file is ~10 KB instead of ~500 MB of full source. Git's patch format is diff-precise — no partial copies, no wrong-file-version issues. The server's git apply verifies the patch applies cleanly or reports a conflict clearly. git reset --hard before applying ensures the server is always in a reproducible state.

5. Remote Build: bee-boost Distributed System

The LGE build infrastructure uses bee-boost — a distributed build farm that parallelizes AOSP compilation across multiple worker nodes. The script triggers it via SSH and monitors the build log:

Python build.py — build monitoring loop 
if IS_BUILD_BOOST == 'YES':
    # Trigger: e.g. "bee-boost-start ~/bee_build/gvm-systemui.sh"
    util.exec(server_ssh, COMMAND_BUILD_BOOST)
    time.sleep(30)     # wait for build to initialize

    is_during_build = True
    while(is_during_build):
        time.sleep(10)
        connect_to_server()   # reconnect if build server rebooted during build
        result = build_result()

        if result == -1:       # "failed to build some targets" in log
            util.error("Build failed! check output/bee_build.log")
            util.showWindowNotification("Error", "Build Failed!!")
            sys.exit()
        elif result >= 1:    # "build completed successfully" found
            util.info("Build success")
            is_during_build = False
        else:                  # still running — stream tail of log
            util.exec(server_ssh, 'tail -f -n100 ~/bee_build/bee_build.log')

def build_result():
    # Download bee_build.log and parse for success/fail markers
    util.make_dir('output')
    os.chdir('output')
    download_from_server("/home/worker/bee_build/", "bee_build.log")

    if util.FindStringInFilePath("failed to build some targets", "bee_build.log") >= 1:
        os.chdir(".."); return -1
    if util.FindStringInFilePath("build completed successfull", "bee_build.log") >= 1:
        os.chdir(".."); return 1
    os.chdir(".."); return 0   # still building

The monitoring loop handles server reboots: the build server may restart mid-build for OS maintenance. The script calls connect_to_server() in every iteration with a 30-retry loop so it can reconnect transparently and continue monitoring without the developer having to restart anything.

6. Download: SFTP Direct + art.lge.com Fallback

The built artefact (e.g., the GvmSystemUI/ folder from AOSP out/) is downloaded with two strategies depending on file size:

Python build.py — download_from_server() 
def download_from_server(path, file_name, use_art_server=False):
    # Always zip first on server (saves bandwidth, handles folders)
    util.exec(server_ssh, f'cd {path} && rm -f compressed.zip && zip -r compressed.zip {file_name}')

    if not use_art_server:
        # Default: direct SFTP download from build server
        util.info('Downloading via SFTP...')
        with server_ftp.cd(path):
            server_ftp.get('compressed.zip', callback=progress_cb)
    else:
        # Large files: server uploads to art.lge.com artifact server first,
        # then Windows client downloads from there (faster CDN, avoids SSH throughput limit)
        util.exec(server_ssh,
            f'cd {path} && curl -u user:pass -T compressed.zip "http://art.lge.com/vc/aivi2_ccc/GVM/compressed.zip"')
        # Stream download from art.lge.com with progress bar
        response = requests.get(url, auth=auth, stream=True)
        with open('compressed.zip', 'wb') as f:
            for chunk in response.iter_content(chunk_size=8192):
                f.write(chunk)

    # Unzip locally, clean up the zip
    util.unzip('compressed.zip')
    util.remove_entry('compressed.zip')
    util.exec(server_ssh, f'cd {path} && rm compressed.zip')
StrategyWhen usedSpeed
Direct SFTP (pysftp)Default — APK or jar (~5–20 MB)~2–5 MB/s through SSH tunnel
art.lge.com (--art flag)Large artefacts (framework.jar ~200 MB, or full system image)~20–50 MB/s from LGE CDN

7. Install to Device

The -i flag runs the install step: remount the device's OverlayFS, push the APK, then execute the module's reset command:

Python build.py — install step 
if IS_INSTALL_TO_BOARD == 'YES':
    os.chdir('output')
    util.adbRemount()                                     # adb shell mount -o rw,remount /system_ext
    util.pushFileToBoard(OUTPUT_FILE_NAME, DEVICE_FILE_PATH)  # adb push GvmSystemUI/ /system_ext/priv-app/
    os.system(COMMAND_PUSH_AND_RESET)                    # e.g. "install-gvm-systemui" bat alias
    os.chdir("..")

COMMAND_PUSH_AND_RESET from the JSON config maps to a bat alias that kills and restarts the SystemUI process:

Batch install-gvm-systemui.bat (called by COMMAND_PUSH_AND_RESET) 
adb wait-for-device
adb shell "killall com.android.systemui"
adb shell "am start-service com.android.systemui/.SystemUIService"
adb shell "am start-service com.android.systemui/.SystemUIService --display 1"

py build.py -p -b -d -i talkback

10. Why This Approach?

Design Rationale

The tool was built to solve a concrete productivity problem: the AOSP build environment cannot run on a Windows laptop, but source editing and IDE tooling (Android Studio) works best on Windows. The build server is a powerful Linux machine but requires manual SSH commands and file transfers. The Python CLI bridges these two worlds with a single command.

Why Python? Python has Paramiko (the de-facto SSH library for automation scripts) and pysftp, making server interaction trivial to code. Python 3.11 runs natively on Windows without additional runtime. Compared to PowerShell or bash-over-SSH, Python provides a clean abstraction for error handling, retry logic, and cross-platform file operations.

Why git patch instead of rsync? The local repo and server repo may be on different git commits (server may be ahead after other engineers' merges). rsync would blindly overwrite files and break the server's git history. The patch approach is git-aware: it applies only the developer's diff, leaving the server's other changes untouched. This also makes the change set visible (git status on the server shows exactly what was applied).
