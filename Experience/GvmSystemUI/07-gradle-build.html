1. The Problem with AOSP-only Build

GvmSystemUI is derived from AOSP's SystemUI and, by default, must be built with the AOSP build system (Soong/Make). This means:

  • A full AOSP checkout (~200 GB) is required to compile a single app
  • One compilation cycle takes 15–30 minutes on the build server
  • No IDE auto-complete for AOSP internal APIs (@hide, IWindowManager, etc.)
  • Unit tests cannot run locally — no JVM test runner in Soong
  • Engineers spend most of their time waiting for remote builds and log parsing

This is a well-known pain point for teams working on AOSP system applications. The solution is to set up a parallel Gradle build for local development while keeping Soong as the final integration build.

2. The Gradle + Soong Hybrid Strategy

1
Developer edits source in Android Studio Full IDE features: auto-complete, refactoring, inline errors — because Gradle resolves AOSP stub JARs.
2
Gradle build locally ./gradlew assembleDebug — builds APK in ~30 seconds. Signs with the platform key. Installable directly on device via adb.
3
sync_from_soong.py / apply_local_to_soong.py Python scripts sync source files between the Gradle workspace and the AOSP source tree on the build server.
4
Full AOSP build on build server Soong compiles the synced sources as part of the platform image for final integration testing and OTA packaging.

3. AOSP Framework Stubs

SystemUI uses many @hide APIs that are not in the public Android SDK — IWindowManager, IStatusBarService, InsetsFrameProvider, internal PowerManager flags, etc. These are compiled against stub JARs extracted from the AOSP build output and placed in the compile/ directory:

Groovy app/build.gradle 
dependencies {
    // Runtime libraries (included in APK)
    implementation fileTree(dir: 'libs', include: ['*.jar', '*.aar'])

    // Compile-only stubs — AOSP internal APIs (@hide)
    // These exist on the device already; NOT bundled in APK
    compileOnly fileTree(dir: 'compile', include: ['*.jar', '*.aar'])
}

compileOnly is critical here. If the stubs were added as implementation, Gradle would try to bundle them in the APK — causing conflicts with the framework classes already loaded in the device's boot classpath. compileOnly provides them only to the compiler (for type resolution and @hide API access) and they are excluded from the final APK.

Key stub files in compile/

FileContains
framework.jarFull Android framework including @hide APIs
services.jarsystem_server classes: WindowManagerService, IStatusBarService
SystemUI-core.jarAOSP SystemUI base classes that GvmSystemUI extends
lge-ui-lib-0.4.1.aarLGE proprietary UI library (in libs/ — runtime)

4. Platform Signing

SystemUI runs with android:sharedUserId="android.uid.systemui" and requires android:protectionLevel="signature" permissions. This means the APK must be signed with the same platform key as the rest of the system image — otherwise the install will be rejected.

Groovy app/build.gradle — signingConfigs 
android {
    signingConfigs {
        platform {
            storeFile file('./cert/platform.jks')
            storePassword "android"
            keyAlias "androiddebugkey"
            keyPassword "android"
        }
    }
    buildTypes {
        release {
            signingConfig signingConfigs.platform
            minifyEnabled false   // cannot shrink SystemUI — breaks framework class loading
        }
        debug {
            signingConfig signingConfigs.platform
        }
    }
}

The platform.jks file contains the vendor's platform signing certificate — extracted from the AOSP build environment. Both debug and release builds use the platform key because the shared UID must match at all times; a debug-key APK cannot install alongside a platform-key system image.

5. Dagger 2 with kapt

GvmSystemUI uses Dagger 2 for dependency injection (consistent with AOSP SystemUI). Dagger 2 is an annotation processor — it reads @Inject, @Module, @Component annotations at compile time and generates the injection boilerplate. The Kotlin Annotation Processing Tool (kapt) bridges Kotlin source files to Java annotation processors:

Groovy app/build.gradle 
apply plugin: 'kotlin-kapt'

dependencies {
    kapt 'com.google.dagger:dagger-compiler:2.53.1'
    annotationProcessor 'com.google.dagger:dagger-compiler:2.53.1'
    // Both kapt and annotationProcessor declared — ensures Java source files
    // in the mixed Java/Kotlin codebase are also processed
}

Kotlin 2.1 (used in this project) has improved kapt compatibility. However, for new pure-Kotlin modules, KSP (Kotlin Symbol Processing) would be preferred over kapt as it is 2× faster. The project uses kapt because AOSP SystemUI's base Dagger setup is Java-based and KSP migration would require significant refactoring of the inherited component graph.

6. Sync Scripts

Three Python scripts in scripts/ manage the two-way sync between the Gradle workspace and the AOSP source tree:

ScriptDirectionPurpose
sync_from_soong.pyAOSP → GradlePull latest base AOSP SystemUI sources from the build server into the local Gradle project
apply_local_to_soong.pyGradle → AOSPPush local changes to the AOSP source tree for the final platform build
sync_identical.pyBothSync files that are byte-for-byte identical between both trees (tracked in identical.json)

The SOONG_ROOT.json file defines the mapping between Gradle source paths and their corresponding AOSP module paths on the build server. comment_rules.json defines code comment patterns that must be stripped or transformed during sync (e.g., Gradle-specific comments that would break Soong builds).

7. Full Build Configuration Overview

Groovy app/build.gradle — key settings 
android {
    namespace 'com.android.systemui'
    compileSdk 36
    archivesBaseName = "GvmSystemUI"      // output: GvmSystemUI-release.apk

    defaultConfig {
        applicationId "com.android.systemui"
        minSdk 33                           // Android 13 minimum
        targetSdk 35                         // Android 15 target
        versionCode 202
        versionName "2.02"
        multiDexEnabled true               // SystemUI exceeds 65k method limit
    }

    buildFeatures {
        dataBinding true                   // Used in some legacy SystemUI views
        viewBinding true                   // Used in new Kotlin components
    }

    compileOptions {
        sourceCompatibility JavaVersion.VERSION_17
        targetCompatibility JavaVersion.VERSION_17
    }
    kotlinOptions {
        freeCompilerArgs = ['-Xjvm-default=all']  // Kotlin interface default methods
        jvmTarget = '17'
    }

    lint {
        // Suppress lint rules that flag internal Android API usage
        disable += ['ProtectedPermissions', 'SystemPermissionTypo', 'Instantiatable']
        abortOnError = false
    }

    packagingOptions {
        // Exclude duplicate R classes generated by AOSP stubs and Gradle
        exclude '**/*.kotlin_module'
        exclude 'META-INF/DEPENDENCIES'
    }
}

8. Why This Approach?

Design Rationale

The Gradle setup was the single most impactful workflow improvement in the project. Before it, a change to a layout XML required a 20-minute build server cycle, a 10-minute flash, and 5 minutes of device verification — 35+ minutes per iteration. After the Gradle setup, the same layout change takes 30 seconds to build, 10 seconds to adb push and restart SystemUI. That is a 70× improvement in feedback speed. The platform signing and AOSP stub approach are the two non-obvious technical prerequisites that make this possible — without them, the APK either fails to install (wrong key) or fails to compile (missing @hide APIs). The Python sync scripts then ensure that local Gradle changes flow cleanly back into the AOSP platform build, maintaining a single source of truth without manual copy-paste.

70× faster iteration

30s Gradle build vs. 30min AOSP build for local development changes.

Full IDE support

Auto-complete, rename refactoring, and inline errors for @hide APIs via stub JARs.

Local unit tests

JUnit tests run in milliseconds on the developer machine with no device required.

AOSP compatible

Python sync scripts ensure Gradle source stays aligned with the final Soong platform build.
