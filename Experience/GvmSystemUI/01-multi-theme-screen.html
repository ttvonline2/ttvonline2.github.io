1. The Problem

SystemUI must run on 3 different physical screens and support 4 brand themes — all from a single APK.

FPI 11.5"

2560 × 1080 px

FPI 20"

3760 × 1088 px — ultra-wide panoramic display.

RSI 1-2 (12.2")

1920 × 1080 px — RSI1 (user 0) and RSI2 (user 10)

Brand themes: Cadillac, Cadillac V-Series, Chevrolet, GMC.

2. Three Screen Sizes, One APK

Android's resource qualifier system resolves the best-matching folder at runtime (AssetManager) based on the available width in dp. The same APK installs on all displays; Android's framework automatically selects the correct layout, dimension, and drawable folders.

DisplayResolutionWidth in dp (approx)Qualifier Folder Resolved
RSI 1-2 (12.2")1920 × 1080 px~1440 dpres/values-w1440dp/
FPI 11.5"2560 × 1080 px~1920 dpres/values-w1920dp/
FPI 20"3760 × 1088 px~2820 dpres/values-w2820dp/

* Default res/ is not used in production. It is only available for developer testing when manually changing device density or resolution. All production displays match a specific -wNNNdp qualifier bucket.

The -wNNNdp qualifier means "available width is at least NNN dp". Android picks the folder with the largest qualifier that still fits.

3. Resource Qualifier Strategy

Two independent axes must be handled: screen size (static, resolved by Android framework) and brand theme (dynamic, resolved at runtime).

Axis 1 — Same theme, different screen size → layout-wXXXXdp / values-wXXXXdp

All display-size-specific resources live in 3 width-qualified folders. Android's framework automatically picks the best match based on available dp width:

Resource TypeQualifier foldersVaries How?
layout/layout-w1440dp/, layout-w1920dp/, layout-w2820dp/ViewGroup sizes, margins, child counts per screen
values/dimens.xmlvalues-w1440dp/, values-w1920dp/, values-w2820dp/Precise dp: card_width, card_height, nav padding — pixel-perfect per panel
values/themes.xmlsame 3 foldersEach file defines all 4 brand themes, referencing screen-local @dimen/ values
drawable/drawable-w1440dp/, drawable-w1920dp/, drawable-w2820dp/Vector assets at different intrinsic sizes
XML values-w1440dp/dimens.xml (RSI 12.2") 
<!-- 12.2 inch Screen Dimensions (1920x1080) -->
<dimen name="card_width">1259.69dp</dimen>
<dimen name="card_height">434.50dp</dimen>
<dimen name="card_margin_start">63.89dp</dimen>
XML values-w2820dp/dimens.xml (FPI 20") 
<!-- 20 inch Screen Dimensions (3760x1088) -->
<dimen name="card_width">1500.88dp</dimen>
<dimen name="card_height">511.76dp</dimen>
<dimen name="card_margin_start">1075.13dp</dimen>

Axis 2 — Same screen, different theme → Runtime style switching

Each values-wXXXXdp/themes.xml defines all 4 brand styles for that screen size. Style differences (colors, fonts, drawables, icons, nav bar dimensions) are handled via Android theme attributes. At runtime:

  1. SystemUIApplication.replaceTheme() calls CalibrationUtils.getBrand() and pushes the correct @StyleRes into ThemeUtils.
  2. NavigationBarFrame constructor explicitly calls context.setTheme(ThemeUtils.getInstance().getTheme()). Other views (e.g. QuickControlBar) are inflated using the same Dagger-scoped @DisplayId LayoutInflater and inherit the already-themed context — no additional setTheme() needed.
  3. All brand attributes (?attr/navigation_background, ?attr/gm_font, ?attr/card_width_theme…) resolve automatically from the active theme.
Java SystemUIApplication.java — replaceTheme() 
// Called at startup after CalibrationUtils is ready
switch (CalibrationUtils.getInstance().getBrand()) {
    case CADILLAC:
        ThemeUtils.getInstance().setCadillacTheme(R.style.SystemUI_CadillacTheme); break;
    case CHEVROLET:
        ThemeUtils.getInstance().setChevyTheme(R.style.SystemUI_ChevyTheme); break;
    case GMC:
        ThemeUtils.getInstance().setGmcTheme(R.style.SystemUI_GMCTheme); break;
    case CADILLAC_V_SERIES:
        ThemeUtils.getInstance().setCadillacVSeriesTheme(R.style.SystemUI_CadillacVSeriesTheme); break;
    default:
        ThemeUtils.getInstance().setDefaultTheme(R.style.SystemUI_CadillacTheme); break;
}
Java NavigationBarFrame.java 
public NavigationBarFrame(Context context, AttributeSet attrs) {
    super(context, attrs);
    context.setTheme(ThemeUtils.getInstance().getTheme()); // brand style injected here
}

How both axes cross: Android resolves the correct values-wXXXXdp/themes.xml (screen axis) → then setTheme() picks the right brand style from that file (brand axis). Result: 3 screens × 4 brands = 12 combinations, zero code duplication.

4. Multi-Vehicle Brand Themes

Four GM brands share the same hardware but require distinct visual styling. Each values-wXXXXdp/themes.xml defines all 4 brand styles; each style inherits from its parent in LgeUiLib (CadillacTheme, ChevyTheme, GMCTheme, cadillacVSeriesTheme) and overrides dimension attributes via @dimen/ — which Android resolves from the matching qualifier bucket.

XML values-w1440dp/themes.xml (excerpt — 4 styles defined per qualifier folder) 
<style name="SystemUI.CadillacTheme" parent="CadillacTheme">
    <item name="navigation_background">@drawable/navigationbar_background_cadillac</item>
    <item name="navigation_bar_button_height">@dimen/navigation_bar_button_height_cadillac</item>
    <item name="card_width_theme">@dimen/card_width</item>  <!-- resolved from values-w1440dp/dimens.xml -->
</style>
<style name="SystemUI.GMCTheme" parent="GMCTheme"> ... </style>
<style name="SystemUI.ChevyTheme" parent="ChevyTheme"> ... </style>
<style name="SystemUI.CadillacVSeriesTheme" parent="cadillacVSeriesTheme"> ... </style>

Wallpaper is also swapped at runtime via GvmUtils.getWallpaper():

Java GvmUtils.java — getWallpaper() 
switch (brand) {
    case CHEVROLET:         return R.drawable.gm_chevy_wallpaper;
    case GMC:               return R.drawable.gm_gmc_wallpaper;
    case CADILLAC_V_SERIES: return R.drawable.gm_cadillac_v_series_wallpaper;
    default:                return R.drawable.gm_cadillac_wallpaper;
}

5. Runtime Detection & Configuration Changes

1
System boots / display connects Android resolves dp width → selects matching -wNNNdp resource bucket automatically.
2
SystemUI starts → replaceTheme() CalibrationUtils.getBrand() reads brand from calibration partition → ThemeUtils stores the correct @StyleRes.
3
NavigationBarFrame inflated → context.setTheme() NavigationBarFrame constructor explicitly calls context.setTheme(ThemeUtils.getInstance().getTheme()) — brand style applied to the @DisplayId Context.
3b
QuickControlBar view inflated — inherits themed context QuickControlBarModule inflates quick_control.xml using the same Dagger-scoped @DisplayId LayoutInflater. No explicit setTheme() call — the inflater's context is already themed because NavigationBarFrame is resolved first in the dependency graph.
4
ConfigurationController propagates changes Any subsequent config change (DPI, density, locale) dispatches to all listeners so UI re-draws correctly.

6. Why This Approach?

Design Rationale

The alternative — hardcoding pixel checks and if/else branches in Java — would scatter layout logic across the codebase, making it brittle and difficult to test. Android's resource qualifier system is the idiomatic, framework-supported mechanism designed exactly for this problem. It keeps display-specific concerns entirely in the resource layer, the Java/Kotlin code stays generic, and adding a new display size only requires a new resource folder — zero Java changes. For an automotive OEM project with long maintenance cycles and multiple hardware configurations, this separation is critical for scalability.

Zero code duplication

All display variants share 100% of the Java/Kotlin logic. Only resource files differ.

Testable

Each resource qualifier folder can be validated independently with Robolectric by setting a config width.

Extensible

Adding a new vehicle display requires only a new -wXXXdp folder — no APK logic change.

OEM compliance

Single APK artifact satisfies the integration team's requirement for one SystemUI across all vehicle lines.
