1. The Problem

The GM vehicle system requires a panel that can appear on top of any running application — navigation, media, phone — when the user presses a hardware or soft button. This is similar to a system-level bottom sheet. It must:

  • Render above third-party apps without requiring them to participate
  • Blur and dim the content behind it to maintain focus
  • Be dismissible by tapping outside the panel (FLAG_WATCH_OUTSIDE_TOUCH)
  • Appear as part of SystemUI — same process, same UID, system privileges

A standard Android Dialog or Fragment cannot achieve this because it is tied to an Activity's window. The solution is to use WindowManager to add a View directly to the window hierarchy as a system overlay.

3. WindowManager Layout Parameters

Every decision in createWindowLayoutParams() is intentional:

Kotlin QuickControlBar.kt — createWindowLayoutParams() · key params 
// Window type — sits above apps, below keyguard
TYPE_STATUS_BAR_SUB_PANEL

// Flags
  FLAG_WATCH_OUTSIDE_TOUCH   // dismiss on outside tap
FLAG_BLUR_BEHIND or FLAG_DIM_BEHIND              // hardware blur + dim

// Private flags
PRIVATE_FLAG_TRUSTED_OVERLAY   // touch-through on transparent regions
SYSTEM_FLAG_SHOW_FOR_ALL_USERS // multi-user automotive context

// Insets / cutout
params.setFitInsetsTypes(0)   // fill all 4 edges
params.token = Binder()          // Activity-independent window token

Key flag decisions

Flag / PropertyWhy it is set
TYPE_STATUS_BAR_SUB_PANELSits in the Z-order directly above app windows and below the status bar itself. Requires SYSTEM_ALERT_WINDOW or system uid.
FLAG_WATCH_OUTSIDE_TOUCHReceives a single ACTION_OUTSIDE event when the user taps anywhere outside the panel view, allowing auto-dismiss.
PRIVATE_FLAG_TRUSTED_OVERLAYTouch events that land on fully-transparent regions of the overlay are forwarded to the window below — preventing invisible tap-jacking.
setFitInsetsTypes(0)The overlay fills all 4 edges including the space reserved by the navigation bar and status bar.
token = Binder()An Activity-independent Binder token. Without an Activity token, the window survives Activity lifecycle events.

4. Show / Hide Lifecycle

1
show() called mWindowManager.addView(mQuickControlBarView, params) — injects the view into the WMS hierarchy.
2
Animate in QuickControlBarAnimator runs scrim fade-in (0→0.3 alpha) + card slide-up + blur radius 0→max over 333 ms.
3
User interacts Panel controls (brightness, volume, HVAC tiles) are live. TransientBarController suppresses nav-bar auto-hide while panel is open.
4
Dismiss triggered User taps outside (ACTION_OUTSIDE) or presses back → hide() called.
5
Animate out Reverse animation (blur → 0, dim → 0, card slide-down) over 167 ms.
6
removeViewFromWindow() mWindowManager.removeView(mQuickControlBarView) — view fully detached, no memory held by WMS.

5. Blur & Scrim Animations

Hardware blur (FLAG_BLUR_BEHIND) blurs all window content rendered below the overlay. This is a SurfaceFlinger-level feature enabled by ro.surface_flinger.supports_background_blur=1 on the device. The blur radius is animated using a ValueAnimator:

Kotlin QuickControlBarAnimator.kt · key highlights 
// Blur radius per display type
mMaxBlurRadius = if (GvmUtils.isFPI11_5Inch()) 50f else 1f

// Animation durations
ANIMATION_DURATION_IN  = 333L
ANIMATION_DURATION_OUT = 167L

// Blur + dim driven by the same ValueAnimator tick
ValueAnimator.ofFloat(startBlur, endBlur).apply {
    layoutParams.blurBehindRadius = animatedValue
    layoutParams.dimAmount        = (animatedValue / maxBlurRadius) * 0.3f
    windowManager.updateViewLayout(view, layoutParams)
}

6. Dagger 2 Integration

The panel is injected as a per-display subcomponent, allowing one QuickControlBar instance per physical display in a multi-display automotive system. The @DisplayId custom scope qualifier differentiates the WindowManager instance and the inflation context so each panel attaches to the correct display.

7. Why This Approach?

Design Rationale

WindowManager overlay is the only viable mechanism for a system panel that must appear above all third-party apps without their cooperation. Activities cannot host such a panel (they are app-scoped). The TYPE_STATUS_BAR_SUB_PANEL type is the exact window type designed for SystemUI overlays — it is the same type used by AOSP's own Quick Settings panel, Volume dialog, and Notification shade. Reusing this pattern ensures correct Z-ordering, touch event routing, and inset handling without needing to patch Window Policy.

No Activity required

The panel lives entirely in the SystemUI process — no Intent launches, no back-stack, instant show/hide.

Hardware-accelerated blur

FLAG_BLUR_BEHIND delegates blur to SurfaceFlinger — zero CPU cost, no screenshot/bitmap manipulation.

Z-order guaranteed

TYPE_STATUS_BAR_SUB_PANEL always stacks above app windows and below the lock screen.
