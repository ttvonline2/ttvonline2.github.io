1. Testing Challenge in SystemUI

SystemUI is a privileged system application that runs directly on top of the Android framework. Its code is deeply coupled to framework classes that cannot be instantiated in a standard JVM test environment:

  • WindowManager, Context, ContentResolver — require an Android runtime
  • Settings.Global.getInt() — reads from a ContentProvider backed by the system_server
  • Handler(Looper.getMainLooper()) — main looper doesn't exist in a JVM test
  • Dagger 2 component graphs — require all dependences to be constructible

The strategy is to use constructor injection for all framework dependencies, so tests can provide mock instances, and to scope tests to pure business logic — state transitions, timer logic, signal routing — while mocking the Android boundary.

2. Test Stack Decisions

JUnit 4

Not JUnit 5 — AOSP / Android Gradle toolchain has stable JUnit 4 integration. AndroidJUnitRunner on-device tests are JUnit 4 based. Consistent across unit and instrumented tests.

Mockito-Kotlin

Wrapper library from org.mockito.kotlin that makes Mockito idiomatic in Kotlin — no raw generics, proper null handling, type-safe argument matchers.

Robolectric (targeted)

Used selectively for tests that need a real Android Context or Resources resolution without a device. Heavier than pure JVM mocking but avoids full instrumented test overhead.

kotlinx-coroutines-test

For components using Kotlin Coroutines — replaces production dispatcher with UnconfinedTestDispatcher for synchronous test execution.

3. Why Mockito-Kotlin over Java Mockito

Raw Java Mockito has several pain points in Kotlin code:

Kotlin Java Mockito in Kotlin — problem 
// Raw Mockito — verbose, type-unsafe, null warning
val mockContext = Mockito.mock(Context::class.java)
Mockito.when(mockContext.getSystemService(anyString())).thenReturn(mockWm)
// 'when' is a Kotlin keyword — must escape: `when`()
Kotlin mockito-kotlin — clean, idiomatic 
import org.mockito.kotlin.*

val mockContext: Context = mock()          // inferred type
val mockWm: WindowManager = mock()

whenever(mockContext.getSystemService(Context.WINDOW_SERVICE))
    .thenReturn(mockWm)

verify(mockWm).addView(any(), any())    // no raw type parameters

Mockito-Kotlin also handles Kotlin's non-null types correctly — any() returns a non-null T instead of null, avoiding NullPointerException in argument matchers.

4. Test Patterns Used

Dependency Injection via Constructor

All testable classes accept their dependencies through the constructor (enforced by Dagger at runtime). In tests, mocks are substituted directly:

Kotlin FpiMinimalModeControllerTest.kt 
class FpiMinimalModeControllerTest {

    private val mockContentResolver: ContentResolver = mock()
    private val mockMinimalModeHelper: MinimalModeHelper = mock()
    private val mockContext: Context = mock()

    private lateinit var controller: FpiMinimalModeController

    @Before
    fun setUp() {
        whenever(mockContext.contentResolver).thenReturn(mockContentResolver)
        controller = FpiMinimalModeController(mockContext, mockMinimalModeHelper)
    }

Argument Capture

Use argumentCaptor() to inspect values passed to mocked dependencies:

Kotlin Example: capture ContentObserver registration 
@Test
fun `registers observer on passenger state key`() {
    val uriCaptor = argumentCaptor<Uri>()
    controller.start()
    verify(mockContentResolver).registerContentObserver(
        uriCaptor.capture(), eq(true), any()
    )
    assertThat(uriCaptor.firstValue.lastPathSegment)
        .isEqualTo("gmremote.state.passenger")
}

Verifying Interactions

Kotlin Example: passenger empty → launchMinimalMode called 
@Test
fun `launches minimal mode when passenger is empty`() {
    // Given: passenger state = 0 (empty)
    whenever(Settings.Global.getInt(mockContentResolver,
             "gmremote.state.passenger", 0)).thenReturn(0)

    // When: trigger passenger state change
    controller.onPassengerStateChanged()

    // Then: minimal mode should launch
    verify(mockMinimalModeHelper).launchMinimalMode()
    verifyNoMoreInteractions(mockMinimalModeHelper)
}

5. Example: MinimalMode State Logic Tests

The controller has multiple state combinations. A parametrized test covers all:

Kotlin FpiMinimalModeControllerTest.kt 
@Test
fun `does NOT launch minimal mode when passenger is present`() {
    whenever(Settings.Global.getInt(any(), eq("gmremote.state.passenger"), any()))
        .thenReturn(1)   // 1 = Present
    controller.onPassengerStateChanged()
    verify(mockMinimalModeHelper).exitMinimalMode()
    verify(mockMinimalModeHelper, never()).launchMinimalMode()
}

@Test
fun `launches when driving starts and passenger is absent`() {
    whenever(Settings.Global.getInt(any(), eq("gmremote.state.passenger"), any()))
        .thenReturn(0)   // Empty
    whenever(Settings.Global.getInt(any(), eq("gmremote.state.driving"), any()))
        .thenReturn(1)   // Driving
    controller.onDrivingStateChanged()
    verify(mockMinimalModeHelper).launchMinimalMode()
}

6. Example: BurnIn Timer Tests

Handler-based timers require control of the Looper. Using ShadowLooper (Robolectric) or manual clock advancement:

Kotlin BurnInTimeoutHandlerTest.kt 
@RunWith(RobolectricTestRunner::class)
class BurnInTimeoutHandlerTest {

    private val mockCallback: BurnInCallback = mock()

    @Before
    fun setUp() = BurnInTimeoutHandler.setCallback(mockCallback)

    @Test
    fun `fires callback after 110 minutes of no reset`() {
        BurnInTimeoutHandler.resetTimer()
        // Advance Robolectric's clock by 110 minutes
        ShadowLooper.idleMainLooper(110 * 60 * 1000, TimeUnit.MILLISECONDS)
        verify(mockCallback).onUserInactivityTimeout()
    }

    @Test
    fun `does NOT fire if reset is called before timeout`() {
        BurnInTimeoutHandler.resetTimer()
        ShadowLooper.idleMainLooper(60 * 60 * 1000, TimeUnit.MILLISECONDS) // 60 min
        BurnInTimeoutHandler.resetTimer()   // restart countdown
        ShadowLooper.idleMainLooper(60 * 60 * 1000, TimeUnit.MILLISECONDS) // another 60 min
        // Total: 120 min elapsed but timer was reset at 60 → only 60 min since last reset
        verify(mockCallback, never()).onUserInactivityTimeout()
    }
}

7. Gradle Test Configuration

The test runner is configured in app/build.gradle:

Groovy app/build.gradle 
android {
    defaultConfig {
        testInstrumentationRunner 'androidx.test.runner.AndroidJUnitRunner'
    }
    testOptions {
        unitTests.all {
            // Enabled in CI; locally disabled for faster builds
            enabled = false
        }
        unitTests {
            includeAndroidResources = true  // Required for Robolectric resource access
        }
    }
}

dependencies {
    testImplementation 'junit:junit:4.13.2'
    testImplementation 'org.mockito.kotlin:mockito-kotlin:5.2.1'
    testImplementation 'org.robolectric:robolectric:4.12.1'
    testImplementation 'org.jetbrains.kotlinx:kotlinx-coroutines-test:1.8.0'
    androidTestImplementation 'androidx.test.ext:junit:1.2.1'
    androidTestImplementation 'androidx.test.espresso:espresso-core:3.6.1'
}

unitTests.all { enabled = false } is set locally to prevent test execution during the main build cycle (which syncs to the build server). Tests are re-enabled in the CI/CD pipeline where a clean environment ensures accurate results. This avoids false failures from missing framework stubs in the dev environment.

8. Why This Approach?

Design Rationale

The project is an automotive system application where bugs can affect safety-critical driver interfaces. Unit tests on state machine logic (MinimalMode trigger conditions, BurnIn timer restart behavior) provide fast regression feedback — running in milliseconds on a developer's machine with no device or emulator required. Mockito-Kotlin was chosen over Kotlin's kotlinx.coroutines.test MockK library because the existing AOSP SystemUI test patterns across the team used Mockito, ensuring consistency. JUnit 4 over JUnit 5 because AndroidJUnitRunner (used for instrumented on-device tests) is JUnit 4-based — using the same framework for both unit and integration tests reduces cognitive overhead and tooling configuration.
