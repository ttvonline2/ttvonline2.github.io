1. Why Blur?

The QuickControl panel overlays the entire screen content. Without any treatment of the content behind it, the result is a harsh opaque overlay that visually disconnects from the running application. Modern automotive UX guidelines (and consumer expectations from iOS/Android) require a frosted-glass depth effect — content below is recognizable but de-emphasized, signaling to the user that the overlay is a temporary layer above their context.

There are three implementation options:

ApproachPerformanceQualityComplexity
CPU screenshot + blur bitmap (RenderScript / CPU)❌ High CPU, ~100–300ms lagStatic, no live updateMedium
RenderEffect blur on a View✅ GPU, but blurs own view not backgroundOnly own contentLow
FLAG_BLUR_BEHIND (SurfaceFlinger blur)✅ GPU, live, 0 ms overheadLive — updates as content below movesLow

The SurfaceFlinger approach is clearly superior for a live overlay — it blurs at the compositor level, meaning even videos playing behind the panel are blurred in real time.

2. Enabling Blur: System Property

Background blur via FLAG_BLUR_BEHIND is disabled by default in AOSP because it requires GPU hardware support and can impact power on low-end devices. It is enabled with a single system property in the device's vendor/build.prop or product/build.prop:

Properties device/gm/gvm/build.prop (or init.rc) 
# Enable SurfaceFlinger hardware background blur
ro.surface_flinger.supports_background_blur=1

Without this property set to 1, SurfaceFlinger ignores blurBehindRadius on all windows. The Qualcomm SA8155 GPU used in the GM vehicle platform supports this feature natively.

You can verify blur support at runtime: BlurUtils.supportsBlursOnWindows(windowManager) wraps an internal API check on this system property. QuickControlBar gates the blur feature on this check to gracefully degrade to a dim-only overlay on platforms without blur support.

3. Window Flags for Blur

Two window parameters work together to produce the frosted-glass effect:

ParameterWhat it controls
FLAG_BLUR_BEHINDEnables Gaussian blur on all layers behind this window's surface in SurfaceFlinger
blurBehindRadiusBlur kernel radius in pixels. 0 = no blur. Animated from 0 to max on show.
FLAG_DIM_BEHINDAdds a dark scrim behind the window (separate from blur)
dimAmountScrim opacity 0.0–1.0. Animated alongside blur for a combined frosted effect.

5. Per-Display Blur Radius Tuning

The blur radius is not the same across all displays because the pixel density differs. A radius of 50px at 2560×1080 produces a very different perceptual blur than 50px at 1920×1080.

The RSI displays use 1f as their blur radius, meaning blur is effectively a no-op there and only the dim effect is active. This is a calibration decision: on the larger RSI panels under bright vehicle cabin lighting, a heavy blur was found to be less legible. The dim-only approach provides sufficient visual separation.

6. Why Not CPU-based Blur?

Design Rationale

The primary reason for choosing FLAG_BLUR_BEHIND is zero CPU cost. CPU-based blur (blurring a screenshot bitmap with RenderScript or custom Gaussian convolution) would take 100–300ms per frame on the ARM cores, making real-time animation impossible and blocking the main thread. SurfaceFlinger's blur happens completely on the GPU's compositor path — it processes the raw surface buffers before they reach the display, adding negligible latency (~1ms). The effect also stays live: if a video is playing full-screen beneath the panel, the blur refreshes at 60 fps as the video frames change.

Enabling it required only a system property and two window flags — the lowest possible implementation cost for a high-quality effect.
