1. Architecture Context

The BMW HKMC project runs on a QNX Hypervisor hosting two completely independent OS containers side-by-side: a WebOS container (the main in-car infotainment) and an Android VM container (for Android apps and services). The two containers communicate over an LCP bus (LGE Container Protocol).

The HMI layer is an Android application (webosprojectionhmi) that must appear visually on top of whatever WebOS is rendering — acting as a transparent glass pane that can show Android popups, overlays, and launchers without blocking the WebOS content underneath.

 WebOS Container

Main car infotainment UI, apps, and services rendered by WebOS.

 Android VM Container

Android apps, HMI overlay, projection service, and system service.

 LCP Bus

LGE's inter-container message bus bridging the two OS worlds.

2. Transparent Compose Activity over WebOS

The key challenge: Android needs a visible window surface on screen, but it must not occlude the WebOS rendering below. The solution is a fully transparent ComponentActivity whose Compose content is an invisible Box — only popup layers rendered over it become visible.

How it works

MainActivity declares itself as a HOME / LAUNCHER Activity so Android treats it as the launcher — it is always present in the window stack. Its Compose tree renders nothing by default (a transparent container), but the ProjectionHmiViewModel can push overlays (popups, screen-share UI) into it on demand. 

// MainActivity.kt — stripped to essentials
@AndroidEntryPoint
class MainActivity : ComponentActivity() {

    @Inject lateinit var viewModel: ProjectionHmiViewModel

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent {
            // Fully transparent root — WebOS renders beneath this surface
            Box(modifier = Modifier
                .fillMaxSize()
                .background(Color.Transparent)) {

                // Overlay content pushed by ViewModel state (popups, screen-share UI)
                val isSharing by viewModel.isInScreenSharingMode.collectAsState()
                if (isSharing) { ScreenShareOverlay() }
            }
        }
        viewModel.activityLifecycleChange("onCreate")
    }

    override fun onStart()   { viewModel.activityLifecycleChange("onStart") }
    override fun onResume()  { viewModel.activityLifecycleChange("onResume") }
    override fun onPause()   { viewModel.activityLifecycleChange("onPause") }
    override fun onStop()    { viewModel.activityLifecycleChange("onStop") }
}

Why Jetpack Compose?

Compose's declarative model makes it trivial to show/hide overlays purely via state — no manual View.setVisibility() juggling. Overlays appear when a StateFlow emits true and vanish when it returns to false.

Why Not a regular View?

Traditional XML layouts require separate inflation, ViewGroup management, and complex animation code. Compose reduces the overlay feature to a single conditional block in the composition.

AndroidManifest: HOME Launcher + transparent theme

<activity
    android:name=".main.MainActivity"
    android:launchMode="singleTask"
    android:theme="@android:style/Theme.Translucent.NoTitleBar.Fullscreen"
    android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.MAIN"/>
        <category android:name="android.intent.category.HOME"/>
        <category android:name="android.intent.category.LAUNCHER"/>
    </intent-filter>
</activity>

singleTask ensures only one instance exists. The translucent fullscreen theme sets the window background to transparent at the framework level — no Compose workaround needed.

3. System UID Privileges

The HMI app is signed with the platform key and declares android:sharedUserId="android.uid.system" — granting it the same UID (1000) as system_server. This is mandatory for several capabilities the app needs:

CapabilityWhy System UID Is Required
INTERACT_ACROSS_USERS Bind to WebosProjectionService as user 0 regardless of current user session.
Launch Activity on specific Display ID ActivityOptions.makeBasic().setLaunchDisplayId() requires elevated privilege.
Bind to protected services lge.wosp.service.PERMISSION_BIND_SERVICE is a signature-level permission.
Window overlay without user prompt System apps can draw overlays without the SYSTEM_ALERT_WINDOW prompt. 
<!-- AndroidManifest.xml -->
<manifest
    package="com.lge.app.car.webosprojectionhmi"
    android:sharedUserId="android.uid.system">

    <uses-permission android:name="android.permission.RECEIVE_BOOT_COMPLETED"/>
    <uses-permission android:name="android.permission.INTERACT_ACROSS_USERS"/>
    <uses-permission android:name="lge.wosp.service.PERMISSION_BIND_SERVICE"/>

    <queries>
        <!-- Required to query & bind the projection service -->
        <intent>
            <action android:name="com.lge.car.webosprojectionservice.IWebosProjectionInterface"/>
        </intent>
    </queries>
</manifest>

4. AIDL Binder for Service Communication

The HMI app communicates with the WebosProjectionService (running in a separate process) via a pair of AIDL interfaces. AIDL generates the Binder stub/proxy boilerplate, giving you a type-safe, process-boundary-crossing RPC call that looks like a regular function call.

Interface design

// IWebosProjection.aidl — Service exposes this
interface IWebosProjection {
    // Register/unregister for push notifications from WebOS
    void registerWebosProjectionListener(in IWebosProjectionListener listener);
    void unRegisterWebosProjectionListener(in IWebosProjectionListener listener);

    // Synchronous read (used for initial state)
    String getWebosData(int msgId);

    // Send a message to WebOS (fire-and-forget)
    void sendWebosData(int msgId, in String data);
}

// IWebosProjectionListener.aidl — HMI app implements this callback
interface IWebosProjectionListener {
    // oneway = non-blocking; caller does not wait for return
    oneway void onWebosDataChanged(int msgId, in String data);
}

Why Two Interfaces?

IWebosProjection is the service stub (server-side). IWebosProjectionListener is the callback stub (client-side). This bidirectional pattern avoids polling — the service pushes changes instantly to all registered listeners.

Why oneway?

Binder calls are synchronous by default — the caller blocks until the callee returns. Adding oneway to the callback makes it asynchronous: the service dispatches the notification and immediately continues, avoiding deadlocks when the HMI app is busy.

HMI side: WebosProjectionManager binding

// WebosProjectionManager.kt (simplified)
@Singleton class WebosProjectionManager @Inject constructor(...) {

    private var binder: IWebosProjection? = null

    private val connection = object : ServiceConnection {
        override fun onServiceConnected(name: ComponentName, service: IBinder) {
            binder = IWebosProjection.Stub.asInterface(service)
            binder?.registerWebosProjectionListener(listener)
            flushPendingMessages()   // deliver queued messages
        }
        override fun onServiceDisconnected(name: ComponentName) {
            binder = null
        }
    }

    private val listener = object : IWebosProjectionListener.Stub() {
        override fun onWebosDataChanged(msgId: Int, data: String) {
            dispatchMessage(msgId, data)   // fan-out to registered IFunctionalHandler
        }
    }

    fun bindService() {
        context.bindServiceAsUser(
            Intent(ACTION_WOSP_SERVICE).setPackage(WOSP_SERVICE_PACKAGE),
            connection, BIND_AUTO_CREATE, UserHandle.of(0))
    }
}

5. Lifecycle & Key Event Forwarding

WebOS needs to know whether the Android HMI overlay is focused and in the foreground (so it can adjust audio routing, focus management, etc.). The MainActivity forwards every Android lifecycle callback as a message through the service chain all the way to WebOS. 

// ProjectionHmiViewModel.kt
@HiltViewModel class ProjectionHmiViewModel @Inject constructor(
    private val wospManager: WebosProjectionManager
) : ViewModel() {

    fun activityLifecycleChange(state: String) {
        wospManager.sendMessage(WebosInternalMessage.HmiLifeCycle(state))
        // → serialized to JSON → IWebosProjection.sendWebosData()
        // → WospHmiProxy → WospFocusManager
        // → WospSystemProxy → LCP Bus → WebOS
    }

    fun dispatchKeyEvent(keyCode: Int, action: Int) {
        wospManager.sendMessage(WebosInternalMessage.KeyEvent(keyCode, action))
    }
}
Key insight: The lifecycle flow travels across three process boundaries: HMI App → WebosProjectionService → ContainerCommunicationService → LCP JNI → WebOS. Each hop uses AIDL, making the cross-process calls transparent to the calling code.

Key events (DPAD, BACK from physical hardware buttons) are intercepted in MainActivity.dispatchKeyEvent() and forwarded via the same message pipeline. On the service side, WospInputProxy handles special hardware keys (Netflix, YouTube, Home) and translates them to WebOS string keys ("DOWN", "UP", etc.).

6. Public API for Third-Party Integration

The WebosprojectionApi module is a standalone Android library (.aar) that any third-party app can embed to communicate with the projection service — without needing to know the internal AIDL interfaces or Hilt DI setup.

API Design

// WebosProjectionApiManager.kt — simplified singleton
class WebosProjectionApiManager private constructor(context: Context) {

    companion object {
        @Volatile private var INSTANCE: WebosProjectionApiManager? = null

        fun getInstance(context: Context) =
            INSTANCE ?: synchronized(this) {
                INSTANCE ?: WebosProjectionApiManager(context).also { INSTANCE = it }
            }
    }

    /** Connect to the projection service */
    fun connect() { bindServiceAsUser(...) }

    /** Disconnect cleanly */
    fun disconnect() { context.unbindService(connection) }

    /** Register for push updates on specific message IDs */
    fun registerListener(msgIds: List<Int>, listener: IWebosProjectionApiListener) { ... }

    /** Send a fire-and-forget message */
    fun sendMessage(messageId: Int, data: String?) { ... }

    /** Synchronous getter for current state */
    fun getMessage(messageId: Int): String? = binder?.getWebosData(messageId)
}

// Listener interface a third-party app implements
interface IWebosProjectionApiListener {
    fun onWebosDataReceived(messageId: Int, data: String?)
}

Why a Separate API Module?

Third-party apps (navigation, media, phone) should not depend on internal HMI or service details. The API module acts as a stable contract — internal refactors do not break external integrators as long as the API surface is unchanged.

Thread Safety

The @Volatile singleton with synchronized double-checked locking ensures that concurrent calls from multiple threads always get the same instance, even before the service is connected. Pending messages are queued and flushed on connect.

7. Why These Choices?

ChoiceAlternatives ConsideredReason Selected
Transparent Compose Activity System overlay window (TYPE_APPLICATION_OVERLAY) Activity gives full lifecycle integration, back-stack, and Compose host without extra permission flow.
AIDL Binder IPC Broadcast intents, Messenger, ContentProvider AIDL gives type-safe, bidirectional, low-latency RPC with Java/Kotlin generated stubs. Essential for sub-100ms event latency in automotive.
System UID (sharedUserId) Privileged app without sharedUserId Cross-user binding and display-specific launch require UID 1000. No other mechanism provides the same capability.
Public API module (.aar) Expose service AIDL directly Stable API boundary. Third parties are shielded from internal refactors. Simplifies versioning and documentation.
Jetpack Compose for overlays XML View inflation per popup Single conditional composition tree — no view inflation, no LayoutInflater, no ViewGroup management. Overlay toggling is a single state change.
