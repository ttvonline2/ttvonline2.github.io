1. Why a Separate System Service?

The LCP (LGE Container Protocol) bus is a native C++ SDK provided by the platform team. There is no pure-Java/Kotlin binding — only a .so shared library and C++ headers. A dedicated service (ContainerCommunicationService, app ID com.lge.car.webossystemservice) was created to:

  • Load and own the native projection-lcp library in a single, well-controlled process.
  • Expose the LCP bus to other Android processes via a stable AIDL interface — so the projection service and any future client never need to link native code directly.
  • Run with elevated system privileges required by the LCP SDK (reads /etc/update/official_version, uses private LGE WiFi APIs).

Why Unix socket? Android and WebOS run in separate Linux containers and cannot share memory or call each other directly. The socket file at /vendor/var/lib/shared/android1/lcpbus.sock sits on a bind-mount shared volume accessible from all containers. lcp-busd owns the socket and acts as a broker — every client connects to it and messages are routed by the registered destination name.

4. ContainerCommunicationStub — the AIDL Bridge

A Hilt-injected singleton that implements IContainerCommunicationInterface.Stub. It coordinates all message flow between AIDL callers and the LCP bus, and listens to system events via DataChangeDispatcher.

Initialization — strict order to prevent message loss

Step 1 — Register callback BEFORE connecting
Communicator.connection.registerMessage(lcpIMessageCallback)
Stores the Kotlin callback in the JNI layer so incoming messages are captured even if they arrive during the connection handshake.
↓
Step 2 — Connect
connectLcp() → Communicator.connection.connect("WEBOS_PROJECTION_HMI")
Registers this process as a named client on lcp-busd. WebOS services can now send messages addressed to "WEBOS_PROJECTION_HMI".
↓
Step 3 — Subscribe to system events
DataChangeDispatcher.registerListener(this)
Receives notifications when system state changes (e.g. WiFi connection).

Two Distinct Inbound Paths

Path A — LCP Bus message (WebOS → Android)

  • lcpIMessageCallback.onCallback(cmd, bytes) fires
  • Parse bytes → JSON string
  • Extract "container_id" from JSON (default: 1)
  • Call triggerMessageCallback(cid, cmd, payload)
  • → Delivered to that specific CID only
  • No callback yet? → store in messageMap[cid] queue

Path B — System event (e.g. WiFi change)

  • SystemNetworkInfoManager detects state change via WifiProxyManager
  • Calls DataChangeDispatcher.notifyDataChanged(cmd, json)
  • → Stub.onDataChanged(cmd, data) fires
  • Loops i = 1 to 2: calls triggerMessageCallback(i, ...)
  • → Broadcast to ALL CIDs (driver + passenger)
Key distinction: LCP bus messages embed a container_id field in their JSON payload — only the matching CID receives the message. System events (network state, version info) have no per-CID scope, so they are broadcast to all active CIDs via the onDataChanged path.

AIDL Interface Methods

MethodDirectionBehavior
registerMessage(cid, cb) Client → Service Stores callback in callbackMap[cid]. Immediately replays any queued messages from messageMap[cid] that arrived before registration.
registerStatus(name, cb) Client → Service Monitors connect/disconnect of a named LCP peer. Notifies via IStatusCallback.
sendMessage(cid, cmd, data) Client → LCP Bus Parses data as JSON, delegates to Communicator.connection.sendMessage(cid, cmd, JSONObject) → JNI → socket write.
getSystemData(cmd) Client → Service (sync) Synchronous read. Routes to SystemSettingInfoManager (version, region, language) or SystemNetworkInfoManager (network state) by cmd.
setSystemData(cmd, data) Client → Service Declared in AIDL — reserved for future use (currently empty implementation).

Why CID routing?

The Android VM supports two concurrent user sessions on separate screens (driver + passenger). Each session has a CID (1 or 2). WebOS embeds the target container_id in message payloads to address a specific session. System-wide events skip this and broadcast to all CIDs.

Pre-registration queue

lcp-busd can deliver messages before the projection service binds and calls registerMessage(). Messages are stored in messageMap[cid] and replayed immediately on registration — zero message loss on startup.

5. System Managers

Two Hilt-injected singletons provide data for getSystemData() queries and push real-time updates through DataChangeDispatcher.

SystemSettingInfoManager

  • getVersionInfo() — reads etc/update/official_version; fallback: hardcoded build string
  • getRegion() — hardcoded "Europa" (I-290 project)
  • getLanguage() — hardcoded "en-US" (I-290 project)
  • Synchronous reads only — no background monitoring

SystemNetworkInfoManager

  • Uses LGE proprietary WifiProxyManager (AIDL-based, not standard Android BroadcastReceiver)
  • Registers IWifiEventListener.Stub() for WiFi events
  • Monitors SUPPLICANT_CONNECTION_CHANGE_ACTION
  • On state change → DataChangeDispatcher.notifyDataChanged()
  • getNetworkOnOff() → JSON: isInternetConnectionAvailable
  • getNetworkInfo() → JSON: type, IP address

Network Event Flow

WifiProxyManager → IWifiEventListener.onWifiEventListener()
LGE AIDL callback fires on SUPPLICANT_CONNECTION_CHANGE_ACTION
↓ Only fires when connected state actually changes (debounced)
DataChangeDispatcher.notifyDataChanged(GET_NETWORK_ONOFF / GET_NETWORK_INFO, json)
↓ pub-sub: all registered SystemDataChangeListeners notified
ContainerCommunicationStub.onDataChanged(cmd, data)
Broadcasts to CID 1 and CID 2 via triggerMessageCallback loop

6. CMake Build for the Native Library

The system module uses CMake (via Android NDK) to compile all JNI bridge .cpp files and link them against the pre-built LCP SDK, producing libprojection-lcp.so.

Source files (auto-collected)

  • All *.cpp under src/main/cpp/ via file(GLOB_RECURSE)
  • JniOnLoad.cpp
  • LcpBusNativeService.cpp
  • LcpBusNativeManager.cpp
  • JniHelper.cpp

Pre-built SDK (IMPORTED)

  • jniLibs/arm64-v8a/libLCPBus.so — real device build
  • jniLibs/arm64-v8a/libLCPBus_window.so — Windows dev environment
  • CMake detects OS=Windows_NT env var to switch automatically
↓ Linked with: lcp_bus · log · android
Output: libprojection-lcp.so
ABI: arm64-v8a only (Qualcomm SA8155) · C++17 · c++_shared STL · Release build

arm64-v8a only

The Qualcomm SA8155 SoC is an ARM64-only platform. A single ABI target reduces APK size and avoids shipping unused x86/armeabi variants on a fixed-hardware automotive system.

c++_shared STL

libLCPBus.so uses the shared C++ STL (libc++_shared.so). Setting -DANDROID_STL=c++_shared ensures both libraries share the same STL instance, preventing duplicate symbol conflicts at runtime.

Windows dev build support

CMake checks the OS environment variable at build time. On Windows, it links libLCPBus_window.so instead of the production binary, enabling local development and compilation without a Linux build host.

7. Why These Choices?

ChoiceAlternative ConsideredReason Selected
Separate system service process Load native library directly in the projection service A native crash in JNI only kills the system service, not the projection UI. Process isolation improves resilience and allows independent restart.
AIDL boundary between processes Local Binder (same-process interface) AIDL enforces a strict API contract, enables independent module deployment, and provides Binder-level UID security so only authorized processes can access the LCP bus.
Register callback BEFORE connecting Connect first, then register lcp-busd can push messages immediately on connection. Registering the Kotlin callback first guarantees no message is dropped during the brief connection setup window.
DataChangeDispatcher pub-sub for system events Direct call from SystemNetworkInfoManager to stub The dispatcher decouples SystemNetworkInfoManager from ContainerCommunicationStub. The manager fires events without knowing who is listening, keeping each component independently testable and maintainable.
Pre-registration message queue Drop messages before registration; require explicit re-fetch Automotive startup timing is tight. Messages from WebOS can arrive milliseconds after boot before the projection service finishes binding. Queuing prevents any silent data loss during the startup race.
LGE WifiProxyManager (not BroadcastReceiver) Standard Android WifiManager + BroadcastReceiver The platform exposes WiFi state through a proprietary AIDL service (WifiProxyManager) which provides finer-grained events and DHCP info not available through the standard Android WiFi API on this automotive platform.
