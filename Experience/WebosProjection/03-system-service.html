1. Why a Separate System Service?

The LCP (LGE Container Protocol) bus is a native C++ SDK provided by the platform team. There is no pure-Java/Kotlin binding — only a .so shared library and C++ headers. Because JNI code is tied to a specific process and privilege level, a dedicated service (ContainerCommunicationService, app ID com.lge.car.webossystemservice) was created to:

  • Load and own the native projection-lcp library in a single well-controlled process.
  • Expose the LCP bus to other Android processes via a stable AIDL interface — so the projection service and any future client never need to link native code directly.
  • Run with elevated system privileges required by the LCP SDK (reads /etc/update/official_version, uses private WiFi APIs).

2. Layer Architecture

The system module has three clean horizontal layers:

① AIDL Boundary Layer

ContainerCommunicationStub — implements IContainerCommunicationInterface.Stub, the public AIDL interface that external clients (projection service) call.

② Kotlin Orchestration Layer

Communicator singleton + LcpCommunicator — route messages between AIDL callers and the native library. Manage multi-CID routing and callback registration.

③ Native JNI Layer

projection-lcp shared library — wraps the LCP C++ SDK. Exposes external funs in LcpCommunicator.kt via System.loadLibrary().

3. JNI Gateway: LcpCommunicator

LcpCommunicator is where Kotlin meets native code. It loads the native projection-lcp library and declares external fun signatures that map directly to JNI functions implemented in C++. 

// LcpCommunicator.kt
class LcpCommunicator : ICommunicator {

    companion object {
        init {
            // Loads libprojection-lcp.so from the APK's lib/ directory
            System.loadLibrary("projection-lcp")
        }

        // JNI externals — implemented in C++ (projection-lcp source)
        external fun connectLCPBus()
        external fun registerCallback(callback: Any)
        external fun registerStatusCallback(name: String, callback: Any)
        external fun sendMessage(managerId: Int, cmd: Int, data: String)
    }

    // Map Android CID (container ID) to the LCP manager slot
    private fun cidToManagerId(cid: Int) = when (cid) {
        1 -> HCC_MANAGER1   // user 0 (main user)
        2 -> HCC_MANAGER2   // secondary user / second display
        else -> throw IllegalArgumentException("Unknown cid: $cid")
    }

    override fun connect() {
        connectLCPBus()   // establishes connection to the LCP daemon
    }

    override fun sendMessage(cid: Int, cmd: Int, data: String) {
        sendMessage(cidToManagerId(cid), cmd, data)
    }

    // Called FROM native code when a message arrives from WebOS
    fun onMessageReceived(cmd: Int, data: String) {
        DataChangeDispatcher.notifyDataChanged(cmd, data)
    }
}
JNI callback direction: The C++ library calls back into Java/Kotlin via a registered JNI callback object. This is the standard Android JNI callback pattern: Java passes a reference to a Kotlin object; the native layer stores a global JNI reference and calls its methods when an event arrives on the LCP bus.

Communicator — the Kotlin singleton wrapper

// Communicator.kt — lazy-initialized Kotlin object
object Communicator {
    val connection: ICommunicator by lazy { LcpCommunicator() }
}

The object singleton with lazy initialization guarantees the native library is loaded exactly once regardless of how many callers reference Communicator.connection.

4. ContainerCommunicationStub — the AIDL Bridge

ContainerCommunicationStub implements IContainerCommunicationInterface.Stub — the Binder stub that remote processes call. It routes every call between the Kotlin/AIDL world and the native LCP layer. 

@Singleton class ContainerCommunicationStub @Inject constructor(
    private val systemInfoMgr:    SystemSettingInfoManager,
    private val systemNetworkMgr: SystemNetworkInfoManager,
) : IContainerCommunicationInterface.Stub() {

    // Registered callbacks per CID (container ID)
    private val msgCallbacks    = ConcurrentHashMap<Int, IContainerCommunicationInterface.IMessageCallback>()
    // Messages that arrived before client registered its callback
    private val pendingMessages = ConcurrentHashMap<Int, Queue<Pair<Int, String>>>()

    init {
        // Register a callback to receive ALL incoming messages from LCP
        Communicator.connection.registerCallback(object {
            fun onReceive(cmd: Int, data: String) {
                // Fan-out to cid 1 AND cid 2
                triggerMessageCallback(1, cmd, data)
                triggerMessageCallback(2, cmd, data)
                DataChangeDispatcher.notifyDataChanged(cmd, data)
            }
        })
        Communicator.connection.connect()
        systemNetworkMgr.init(DataChangeDispatcher)
    }

    // Called by WospSystemProxy.bind() over AIDL
    override fun registerMessage(cid: Int, cb: IMessageCallback): Boolean {
        msgCallbacks[cid] = cb
        // Flush any messages that arrived before registration
        pendingMessages[cid]?.forEach { (cmd, data) -> cb.onCallback(cmd, data) }
        pendingMessages.remove(cid)
        return true
    }

    // Called by WospSystemProxy when sending data TO WebOS
    override fun sendMessage(cid: Int, cmd: Int, data: String): Boolean {
        Communicator.connection.sendMessage(cid, cmd, data)
        return true
    }

    // Synchronous data getter (version, region, language, network)
    override fun getSystemData(cmd: Int): String = when (cmd) {
        GET_VERSION_INFO  -> systemInfoMgr.getVersionInfo()
        GET_REGION        -> systemInfoMgr.getRegion()
        GET_LANGUAGE      -> systemInfoMgr.getLanguage()
        GET_NETWORK_ONOFF -> systemNetworkMgr.getNetworkOnOff()
        GET_NETWORK_INFO  -> systemNetworkMgr.getNetworkInfo()
        else -> "{}"
    }

    private fun triggerMessageCallback(cid: Int, cmd: Int, data: String) {
        val cb = msgCallbacks[cid]
        if (cb != null) cb.onCallback(cmd, data)
        else pendingMessages.getOrPut(cid) { LinkedList() }.add(cmd to data)
    }
}

Why CID routing?

The Android VM supports two concurrent user sessions displayed on separate screens (driver + passenger). Each session is a "container" with its own CID (1 or 2). WebOS messages are broadcast to both CIDs so each user's HMI receives relevant events.

Pre-registration queue

The LCP bus can deliver messages before the projection service has bound and called registerMessage(). A ConcurrentHashMap of pending queues stores these messages and replays them immediately upon registration — zero message loss on startup.

5. System Managers

The system module contains lightweight managers that serve synchronous data queries (called from getSystemData()).

SystemSettingInfoManager

@Singleton class SystemSettingInfoManager {
    fun getVersionInfo(): String {
        // Read OTA version from platform file
        return try {
            File("/etc/update/official_version").readText().trim()
        } catch (e: Exception) { "1.0.0.0000" }
    }
    fun getRegion(): String   = "Europa"
    fun getLanguage(): String = "en-US"
}

SystemNetworkInfoManager

@Singleton class SystemNetworkInfoManager @Inject constructor(@ApplicationContext ctx: Context) {

    private val wifiMgr = WifiProxyManager(ctx)   // LGE-internal WiFi API

    fun init(dispatcher: DataChangeDispatcher) {
        // Monitor supplicant DHCP state changes
        ctx.registerReceiver(object : BroadcastReceiver() {
            override fun onReceive(ctx: Context, intent: Intent) {
                dispatcher.notifyDataChanged(GET_NETWORK_ONOFF, getNetworkOnOff())
                dispatcher.notifyDataChanged(GET_NETWORK_INFO,  getNetworkInfo())
            }
        }, IntentFilter(WifiManager.SUPPLICANT_CONNECTION_CHANGE_ACTION))
    }

    fun getNetworkOnOff(): String =
        Gson().toJson(mapOf("isInternetConnectionAvailable" to wifiMgr.isConnected))

    fun getNetworkInfo(): String {
        val dhcp = wifiMgr.dhcpInfo
        return Gson().toJson(mapOf(
            "type"                         to "wifi",
            "isInternetConnectionAvailable" to wifiMgr.isConnected,
            "wifi"                         to mapOf("ipAddress" to dhcp.ipAddress)
        ))
    }
}

6. CMake Build for the Native Library

The system module's build.gradle.kts configures the NDK and CMake to build the projection-lcp shared library that wraps the LCP SDK. 

// system/build.gradle.kts (relevant parts)
android {
    defaultConfig {
        ndk { abiFilters += setOf("arm64-v8a") }   // Qualcomm SA8155 is ARM64

        externalNativeBuild {
            cmake {
                cppFlags("-std=c++17")
                arguments("-DANDROID_STL=c++_shared")
            }
        }
    }

    externalNativeBuild {
        cmake {
            path = file("src/main/cpp/CMakeLists.txt")
            version = "3.22.1"
        }
    }

    buildFeatures { aidl = true }
}

dependencies {
    // Pre-built LCP SDK JARs and AARs from the platform
    implementation(fileTree(mapOf("dir" to "libs", "include" to listOf("*.jar", "*.aar"))))
}
# CMakeLists.txt (conceptual structure)
cmake_minimum_required(VERSION 3.22.1)
project(projection-lcp)

add_library(projection-lcp SHARED
    lcp_bridge.cpp      # JNI bridge functions
    lcp_communicator.cpp
)

target_link_libraries(projection-lcp
    android log
    ${LCP_SDK_LIB}      # Pre-built LCP C++ SDK .so
)

arm64-v8a only

The Qualcomm SA8155 SoC is an ARM64 platform. Limiting to a single ABI reduces APK size and avoids shipping unused x86/armeabi variants on a fixed-hardware automotive target.

c++_shared STL

The LCP SDK uses the shared C++ STL (libc++_shared.so). Using c++_shared avoids duplicate STL symbols when the LCP .so and the JNI bridge are loaded in the same process.

7. Why These Choices?

ChoiceAlternatives ConsideredReason Selected
Separate system service process Load native library directly in the projection service A native crash in the JNI layer would only kill the system service, not the projection service or HMI. Process isolation improves resilience.
AIDL boundary between service and system Local Binder (same-process interface) AIDL enforces a strict API contract between the two modules, allows independent deployment (system service can be updated without recompiling the projection service), and provides Binder-level security (callers are verified by UID).
JNI in Kotlin companion object (external fun) Wrap in a separate Java utility class Kotlin external fun in a companion object is idiomatic, avoids static Java boilerplate, and allows the JNI function to live alongside its Kotlin counterpart for readability.
DataChangeDispatcher pub-sub for inbound messages Direct callback from JNI to stub The dispatcher decouples the JNI layer from the stub — the JNI callback fires events into the dispatcher, and any number of listeners (stub, network manager) subscribe independently.
Pending message queue (pre-registration) Drop messages before registration; require explicit re-request Automotive startup involves tight timing windows. Messages from WebOS can arrive milliseconds after boot before Android services finish binding. Queueing prevents any silent data loss during the startup race.
