1. Service Overview

WebosProjectionService is a long-lived android.app.Service started at boot (via WebosProjectionBootReceiver) and kept alive with START_STICKY. It owns no UI of its own — its sole job is to route messages between:

 HMI App

Binds via IWebosProjection AIDL. Sends lifecycle, key events, and overlay commands.

 System Service

Bound via IContainerCommunicationInterface AIDL. Bridges to the LCP / WebOS bus.

 Android Platform

Audio, Bluetooth, Network, Input — each wrapped in a dedicated Proxy.

// WebosProjectionService.kt (simplified)
@AndroidEntryPoint
class WebosProjectionService : Service() {

    // Proxies — wrap external data sources
    @Inject lateinit var mWospSystemProxy:    WospSystemProxy

    @Inject lateinit var mWospNetworkProxy:    WospNetworkProxy
    @Inject lateinit var mWospBluetoothProxy:  WospBluetoothProxy
    @Inject lateinit var mWospAudioProxy:      WospAudioProxy
    @Inject lateinit var mWospInputProxy:      WospInputProxy
    @Inject lateinit var mWospCrossAppProxy:   WospCrossAppProxy

    // Managers — orchestrate business logic
    @Inject lateinit var mWospFocusManager:    WospFocusManager
    @Inject lateinit var mWospNetworkManager:  WospNetworkManager
    @Inject lateinit var mWospMediaManager:    WospMediaManager
    @Inject lateinit var mWospBluetoothManager: WospBluetoothManager
    @Inject lateinit var mWospInputManager:    WospInputManager
    @Inject lateinit var mWospCrossAppManager: WospCrossAppManager
    @Inject lateinit var mWospSystemInfoManager: WospSystemInfoManager

    override fun onBind(intent: Intent) = mWospHmiProxy.wospServiceImpl  // expose AIDL binder

    override fun onCreate() {
        super.onCreate()
        mWospSystemProxy.bind()   // connect to system service (with auto-reconnect)
    }

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int) = START_STICKY
}

All 7 proxies and 7 managers are @Singleton Hilt bindings — they are created once, live for the lifetime of the service process, and injected wherever needed.

2. The Proxy Pattern

Every external data source is encapsulated in a Proxy class that extends WospBaseProxy. The proxy is the only place in the codebase that talks directly to its corresponding Android platform API — everything else communicates through WospBaseProxy's listener subscription model.

WospBaseProxy — the subscription engine

// WospBaseProxy.kt (simplified)
abstract class WospBaseProxy {

    // Registry: msgId → list of interested listeners
    private val listenerMap = mutableMapOf<Int, MutableList<WospProxyDataListener>>()

    /** Managers call this in their init{} to subscribe to specific message IDs */
    fun registerMessageCallbackListener(ids: List<Int>, listener: WospProxyDataListener) {
        ids.forEach { id ->
            listenerMap.getOrPut(id) { mutableListOf() }.add(listener)
        }
    }

    /** Proxy calls this internally when new data arrives from the platform */
    protected fun dispatchToProxyListener(msgId: Int, data: WospMsg) {
        listenerMap[msgId]?.forEach { it.onProxyDataChange(msgId, data) }
    }
}

// The listener interface managers implement
interface WospProxyDataListener {
    fun onProxyDataChange(msgId: Int, data: WospMsg)
}

Example: WospNetworkProxy

@Singleton class WospNetworkProxy @Inject constructor(@ApplicationContext ctx: Context)
    : WospBaseProxy() {

    private val connectivityMgr = ctx.getSystemService(ConnectivityManager::class.java)

    private val networkCallback = object : ConnectivityManager.NetworkCallback() {
        override fun onAvailable(network: Network) {
            // Notify all subscribers that network is UP
            dispatchToProxyListener(
                WospMsgID.NetworkGetStatusOnOff,
                NetworkGetStatusOnOff(isInternetConnectionAvailable = true)
            )
        }
        override fun onLost(network: Network) {
            dispatchToProxyListener(
                WospMsgID.NetworkGetStatusOnOff,
                NetworkGetStatusOnOff(isInternetConnectionAvailable = false)
            )
        }
    }

    init {
        connectivityMgr.registerDefaultNetworkCallback(networkCallback)
    }
}

3. The Manager Pattern

Each Manager owns a specific domain of business logic. In its init{} block it subscribes to one or more proxies for the message IDs it cares about — then reacts to those messages and routes responses back through the appropriate proxy.

Anatomy of a Manager (WospFocusManager)

@Singleton class WospFocusManager @Inject constructor(
    private val mWospSystemProxy: WospSystemProxy,
    private val mWospHmiProxy:    WospHmiProxy,
) {
    private var isFocus = false
    private var isForeground = false

    // ── Step 1: declare which message IDs to observe ───────────────────
    // From WebOS (via system proxy)
    private val systemMsgIds = listOf(
        WospMsgID.HccContainerRequestWebosScreenFocusValue,
        WospMsgID.HccContainerRequestWebosForegroundState
    )
    // From HMI app (via HMI proxy)
    private val hmiMsgIds = listOf(WospMsgID.HmiNotifyLifecycle)

    // ── Step 2: define the callback logic ─────────────────────────────
    private val systemListener = WospProxyDataListener { msgId, data ->
        when (msgId) {
            WospMsgID.HccContainerRequestWebosScreenFocusValue ->
                mWospSystemProxy.sendWebosData(
                    WospMsgID.HccHostRespondWebosScreenFocus,
                    HccHostRespondWebosScreenFocus(iFocused = isFocus))

            WospMsgID.HccContainerRequestWebosForegroundState ->
                mWospSystemProxy.sendWebosData(
                    WospMsgID.HccHostRespondWebosForegroundState,
                    HccHostRespondWebosForegroundState(isForeground = isForeground))
        }
    }

    private val hmiListener = WospProxyDataListener { _, data ->
        val lifecycle = data as HmiNotifyLifecycle
        isFocus      = lifecycle.state in listOf("onResume", "onStart")
        isForeground = lifecycle.state in listOf("onResume", "onStart")
    }

    // ── Step 3: register subscriptions ────────────────────────────────
    init {
        mWospSystemProxy.registerMessageCallbackListener(systemMsgIds, systemListener)
        mWospHmiProxy.registerMessageCallbackListener(hmiMsgIds, hmiListener)
    }
}

4. Full Message Flow

Tracing a concrete example — WebOS asks for the device's network status:

① LCP Bus → JNI

WebOS container sends HccContainerRequestNetworkInfo over LCP bus.

② JNI → ContainerCommunicationStub

Native JNI callback fires, triggerMessageCallback(cid=1, cmd=2003, data="{}").

③ AIDL → WospSystemProxy

IContainerCommunicationInterface.IMessageCallback.onCallback() received in service; dispatched to WospSystemProxy.dispatchToProxyListener().

④ WospNetworkManager reacts

Subscribed to HccContainerRequestNetworkInfo; fetches info from WospNetworkProxy and calls mWospSystemProxy.sendWebosData(HccHostRespondNetworkInfo(...)).

⑤ Response goes back

JSON-serialized response travels: WospSystemProxy → IContainerCommunicationInterface.sendMessage() → LCP JNI → LCP Bus → WebOS.

5. Coroutine-Based Auto-Reconnection

The WebosProjectionService binds to ContainerCommunicationService (in the system module, a separate process). That connection can be lost at any time — system service crash, system update, low-memory kill. The service must recover automatically without losing queued messages.

Implementation in WospSystemProxy

@Singleton class WospSystemProxy @Inject constructor(...) : WospBaseProxy() {

    private var systemBinder: IContainerCommunicationInterface? = null
    private var bindAttempts = 0
    private val RETRY_DELAY_MS   = 500L
    private val MAX_RETRY        = 5

    // Messages that arrived before connection was ready
    private val pendingQueue = ConcurrentLinkedQueue<Pair<Int, String>>()

    private val connection = object : ServiceConnection {

        override fun onServiceConnected(name: ComponentName, service: IBinder) {
            bindAttempts = 0
            systemBinder = IContainerCommunicationInterface.Stub.asInterface(service)
            systemBinder?.registerMessage(UserInfoHelper.cid, msgCallback)
            // Flush messages queued while disconnected
            while (pendingQueue.isNotEmpty()) {
                pendingQueue.poll()?.let { (cmd, data) ->
                    systemBinder?.sendMessage(UserInfoHelper.cid, cmd, data)
                }
            }
        }

        override fun onServiceDisconnected(name: ComponentName) {
            systemBinder = null
            if (bindAttempts < MAX_RETRY) scheduleRebind()
        }
    }

    private fun scheduleRebind() {
        bindAttempts++
        // Coroutine delay — non-blocking, no Handler boilerplate
        CoroutineScope(Dispatchers.IO).launch {
            delay(RETRY_DELAY_MS * bindAttempts)   // linear back-off
            withContext(Dispatchers.Main) { bind() }
        }
    }

    fun sendWebosData(msgId: Int, msg: WospMsg) {
        val json = WospMsgConverter.convertWospMsgToJsonString(msg)
        if (systemBinder == null) {
            pendingQueue.add(msgId to json)   // queue until reconnected
        } else {
            systemBinder?.sendMessage(UserInfoHelper.cid, msgId, json)
        }
    }
}

Why Coroutines over Handler.postDelayed?

Handler.postDelayed() requires a Looper and manual cancellation. A coroutine delay() is non-blocking, cancellable, and composable with other async logic. The CoroutineScope(Dispatchers.IO) keeps the retry off the main thread.

Linear Back-off

Retry delay grows linearly (500ms × attempt): 500ms, 1s, 1.5s... up to 5 attempts. This avoids storming the system service with rapid reconnect attempts immediately after a crash.

Message Queue

A ConcurrentLinkedQueue buffers outgoing messages during disconnection. On reconnect, the queue is flushed in order — no messages are lost even during a system service restart.

6. Dependency Injection with Hilt

All proxies and managers are @Singleton Hilt-managed instances. This ensures a single shared instance across the process, eliminates manual lifecycle management, and makes unit testing straightforward — Hilt can replace any proxy with a fake during tests. 

// ServiceApplication.kt — entry point
@HiltAndroidApp
class ServiceApplication : Application()

// WebosProjectionService.kt — injection target
@AndroidEntryPoint
class WebosProjectionService : Service() {
    @Inject lateinit var mWospSystemProxy: WospSystemProxy
    // ... injected lazily by Hilt on onCreate()
}

Because managers subscribe to proxies in their init{} blocks — and Hilt constructs all singletons when the service starts — all subscriptions are wired up before the first message can arrive. There is no initialization ordering problem.

7. Why These Choices?

ChoiceAlternatives ConsideredReason Selected
Proxy/Manager separation Monolithic service class with all logic inline Clear separation of concerns: proxies know how to get data; managers know what to do with it. Each class is under 200 lines and independently testable.
Subscription model (WospBaseProxy) Direct method calls from proxy to manager Adding a new manager never requires changing proxy code. A manager subscribes to many proxies without creating circular dependencies.
Coroutine back-off reconnect AlarmManager, JobScheduler, Handler.postDelayed() Coroutines are lightweight, cancellable, and integrate naturally with Kotlin. delay() suspends without blocking threads.
START_STICKY service Foreground service with persistent notification Automotive head units do not have a traditional notification shade. START_STICKY ensures the system restarts the service after any kill without user-visible notifications.
Hilt for DI Manual singleton pattern, Koin Hilt is the official Android DI framework; integrates natively with @AndroidEntryPoint and produces compile-time validated dependency graphs.
