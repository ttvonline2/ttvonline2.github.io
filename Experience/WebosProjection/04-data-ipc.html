1. The IPC Challenge

The system spans three separate Android processes and a native C++ bus — all needing to exchange structured data objects:

webosprojectionhmi

HMI app process — sends lifecycle, key events, overlay commands.

webosprojectionservice

Service process — central broker, all message routing logic.

webossystemservice

System service process — owns JNI, bridges to native LCP bus.

A Binder/AIDL call can only transfer primitive types and Parcelables natively. Sending 50+ different structured message types across process boundaries without a separate Parcelable class for each one requires a different approach. The solution: serialize every message to a JSON string at the AIDL boundary, and deserialize back to the correct typed class on the receiving side using the message ID as a discriminator.

2. AIDL Standardized Across All Layers

Both AIDL boundaries use the same wire format: (int msgId, String jsonData). This makes the two interfaces composable — a message can be forwarded from one interface to the next without any transformation, just a re-dispatch.

Interface 1: HMI app → Projection Service

// IWebosProjection.aidl
interface IWebosProjection {
    void registerWebosProjectionListener(in IWebosProjectionListener listener);
    void unRegisterWebosProjectionListener(in IWebosProjectionListener listener);
    String getWebosData(int msgId);
    void sendWebosData(int msgId, in String data);   // <-- (msgId, JSON)
}

Interface 2: Projection Service → System Service

// IContainerCommunicationInterface.aidl
interface IContainerCommunicationInterface {
    interface IMessageCallback {
        void onCallback(int cmd, String msg);          // <-- same (msgId, JSON)
    }
    boolean sendMessage(int cid, int cmd, String data);  // <-- same wire format
    String  getSystemData(int cmd);
    // ...registration methods
}
Uniform wire format benefit: Because both AIDL interfaces share (int, String), a message arriving from WebOS via the system service can be forwarded to the HMI app via the projection service without any deserialization in the middle layer — simply relay (cmd, data) as-is. Deserialization only happens at the final consumer.

3. WospMsg — the Type Hierarchy

All messages implement a single marker interface, WospMsg. Each concrete message is a Kotlin data class with exactly the fields defined in the protocol spec. 

// WospMsg.kt
interface WospMsg {
    val msgId: Int
}

// Example: a fragment of WospSystemMsg.kt
data class HccContainerReportLaunchApp(
    val id: String,
    val returnValue: Boolean
) : WospMsg {
    override val msgId = WospMsgID.HccContainerReportLaunchApp   // 1001
}

data class HccHostRespondNetworkInfo(
    val wifi: WifiInfo?,
    val cellular: CellularInfo?,
    val type: String,
    val isInternetConnectionAvailable: Boolean
) : WospMsg {
    override val msgId = WospMsgID.HccHostRespondNetworkInfo   // 2503
}

// Fallback for unknown message IDs
data class UnknownMessage(override val msgId: Int = -1) : WospMsg

The WospMsg interface is a lightweight contract — it only requires msgId. No Parcelable implementation, no Binder-specific boilerplate. Each message class is a pure Kotlin data class serializable with Gson out of the box.

4. Message ID as Type Discriminator

Because JSON itself carries no type information beyond field names, a separate mechanism is needed to know which Kotlin class to deserialize into. The msgId integer already present on the wire serves as the type discriminator.

WospMsgID — the master enum

// KeyMessageID.kt (enum object WospMsgID)
object WospMsgID {
    // HCC messages: WebOS ↔ Android container
    const val HccContainerReportLaunchApp           = 1001
    const val HccContainerNotifyBootComplete        = 1003
    const val HccContainerCommandAudioControl       = 1004
    const val HccContainerRequestWebosScreenFocusValue = 1031
    const val HccHostRespondWebosScreenFocus        = 1531
    const val HccContainerRequestNetworkOnOff       = 2002
    const val HccHostNotifyNetworkOnOff             = 2502
    // ... 40+ more message IDs

    // HMI internal messages
    const val HmiNotifyLifecycle    = 90102
    const val HmiNotifyKeyInformation = 90103

    // System service data commands
    const val SysGetVersionInfo     = 101
    const val SysGetNetworkOnOff    = 102
    const val SysGetRegion          = 103
    const val SysGetLanguage        = 104
    const val SysGetNetworkInfo     = 105
}

The ID ranges reveal the message origin: 1xxx–4xxx = HCC (inter-container protocol messages to/from WebOS); 9xxxx = HMI internal messages; 1xx = system service queries; 3xxxx = Bluetooth; 5xxxx = network.

5. WospMsgConverter — Polymorphic Deserialization

WospMsgConverter is a singleton that holds the registry mapping every msgId to its corresponding Kotlin class, and performs the actual Gson deserialization. 

// WospMsgConverter.kt
object WospMsgConverter {

    // ── Type registry: msgId → Kotlin data class ──────────────────────────
    private val msgTypeMap: Map<Int, Class<out WospMsg>> = mapOf(
        WospMsgID.HccContainerReportLaunchApp      to HccContainerReportLaunchApp::class.java,
        WospMsgID.HccContainerCommandAudioControl  to HccContainerCommandAudioControl::class.java,
        WospMsgID.HccHostRespondNetworkInfo        to HccHostRespondNetworkInfo::class.java,
        WospMsgID.HmiNotifyLifecycle               to HmiNotifyLifecycle::class.java,
        WospMsgID.NetworkGetStatusOnOff            to NetworkGetStatusOnOff::class.java,
        WospMsgID.BluetoothDeviceStateChanged      to BluetoothDeviceStateChanged::class.java,
        // ... all 50+ message types
    )

    private val gson = Gson()

    // ── Deserialization: JSON → WospMsg ───────────────────────────────────
    fun convertJsonStringToWospMsg(msgId: Int, json: String): WospMsg {
        val clazz = msgTypeMap[msgId] ?: return UnknownMessage(msgId)
        return gson.fromJson(json, clazz) ?: UnknownMessage(msgId)
    }

    // ── Serialization: WospMsg → JSON ─────────────────────────────────────
    fun convertWospMsgToJsonString(wospMsg: WospMsg): String {
        // Exclude msgId from the payload — it travels separately as the int parameter
        val exclusion = ExclusionStrategy {
            it.name == "msgId"
        }
        return GsonBuilder()
            .addSerializationExclusionStrategy(exclusion)
            .create()
            .toJson(wospMsg)
    }
}

How deserialization is used in WospHmiProxy

// WospHmiProxy.kt — the HMI-facing AIDL stub
val wospServiceImpl = object : IWebosProjection.Stub() {

    override fun sendWebosData(msgId: Int, data: String) {
        // Step 1: reconstruct the typed message from (msgId, JSON)
        val wospMsg = WospMsgConverter.convertJsonStringToWospMsg(msgId, data)

        // Step 2: dispatch to all listeners registered for this msgId
        dispatchToProxyListener(msgId, wospMsg)
        // → managers receive a WospMsg and cast to the expected subtype
    }
}

Why msgId outside the JSON?

Keeping msgId as a separate AIDL parameter (not embedded in the JSON body) means the routing layer can read the ID without parsing JSON at all — a cheap integer comparison routes the message before any deserialization work happens.

Why Gson over Kotlinx-Serialization?

The project targets Android SDK 33-35 and the LGE toolchain already ships Gson. More importantly, Gson's reflection-based deserialization works directly with the existing data classes with zero additional annotations — Kotlinx Serialization would require @Serializable on every class across hundreds of message types.

6. End-to-End Data Flow

Tracing a message from HMI app to WebOS container:

// ① HMI app creates a typed message and sends it
wospManager.sendMessage(WebosInternalMessage.HmiLifeCycle(state = "onResume"))
// → serialized: msgId=90102, json='{"state":"onResume"}'
// → IWebosProjection.sendWebosData(90102, '{"state":"onResume"}')

// ② WospHmiProxy receives (in service process)
WospMsgConverter.convertJsonStringToWospMsg(90102, '{"state":"onResume"}')
// → returns HmiNotifyLifecycle(state="onResume") : WospMsg
// → dispatched to WospFocusManager via subscription

// ③ WospFocusManager reacts, sends response to WebOS
mWospSystemProxy.sendWebosData(
    WospMsgID.HccHostRespondWebosScreenFocus,
    HccHostRespondWebosScreenFocus(iFocused = true)
)
// → WospMsgConverter serializes: '{"iFocused":true}'
// → IContainerCommunicationInterface.sendMessage(cid=1, cmd=1531, '{"iFocused":true}')

// ④ ContainerCommunicationStub forwards to native
Communicator.connection.sendMessage(cid=1, cmd=1531, '{"iFocused":true}')
// → LcpCommunicator.sendMessage(HCC_MANAGER1, 1531, '{"iFocused":true}')
// → JNI native call → LCP bus → WebOS container
The entire round trip across 3 processes and 1 JNI boundary uses the single uniform wire format: (int msgId, String json). No special Parcelables, no protobuf, no custom serialization format — just JSON over AIDL Binder.

7. Why These Choices?

ChoiceAlternatives ConsideredReason Selected
JSON over AIDL (int, String) Individual Parcelable per message type; Protobuf; custom binary format 50+ message types → 50 Parcelable classes is unmaintainable. JSON requires zero boilerplate per type. Protobuf requires schema files and code generation. JSON is human-readable for debugging.
msgId as type discriminator (outside JSON) Embed a "type" field inside the JSON body Routing decisions need the type before parsing the body. An integer header check is O(1) and avoids a full JSON parse just to determine which manager to route to.
Static registry map in WospMsgConverter Reflection-only deserialization; annotation-based polymorphism (GsonTypeAdapterFactory) An explicit map is compile-time verifiable — missing a message type is a compile warning, not a silent runtime null. It also documents every supported message in one place.
WospMsg marker interface (not sealed class) Sealed hierarchy per domain Messages span multiple files (system, HMI, audio, Bluetooth). A sealed class requires all subtypes in the same file. An interface allows the hierarchy to span files while still providing a common type for the registry and dispatcher.
Same AIDL wire format on both interfaces Different interfaces with typed parameters Forwarding messages between layers requires zero transformation — just relay (cmd, data). Reduces the chance of conversion bugs at layer boundaries.
