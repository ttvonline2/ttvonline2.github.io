1. Why SELinux Matters for FoD

FoD parameter data lives in the persist partition: /mnt/product/persist-alliance/fod/. This directory is not wiped on OTA updates, so it stores vehicle feature entitlement data (e.g. "heated seats: enabled") that the OEM sells to the customer.

Without an explicit SELinux policy, any process running on the Android Automotive system could — intentionally or through a vulnerability — read or overwrite those files:

  • A rogue third-party app could read which premium features the customer has activated.
  • A compromised media-player process could overwrite fod_data and disable paid features.
  • An untested system daemon could corrupt fod_state, stalling the execution pipeline.

The solution: define a unique SELinux type for the FoD directory and grant read/write permission exclusively to the car-service process domain.

2. A 2-Minute SELinux Primer

SELinux implements Mandatory Access Control (MAC): even if a process runs as root, it still needs an explicit allow rule in policy to access a resource. Android Automotive enforces SELinux in Enforcing mode — denials generate an avc: denied log entry and the syscall is blocked at kernel level.

 Type

A label attached to a process (domain) or a file (file type). E.g. nissan_fod_file.

 Allow Rule

allow <domain> <type>:<class> <permissions>; — grants a specific permission set.

 File Context

Maps a filesystem path pattern to a type. The kernel applies this label when the file is created.

 Default Deny

Any access without an explicit allow is denied. No whitelist = blocked.

3. The Threat Model

The FoD directory must satisfy these access-control properties:

Actor Read Write Create/Delete
alliancecarservice_app (FoD service process) ✅ YES ✅ YES ✅ YES
Any other system service (HVAC, Seat, Settings…) ❌ DENIED ❌ DENIED ❌ DENIED
Third-party apps (any UID) ❌ DENIED ❌ DENIED ❌ DENIED
ADB shell (during development, shell domain) ❌ DENIED (enforcing) ❌ DENIED ❌ DENIED

4. Defining the Type: file.te

The first step is declaring the new type in the SELinux policy source. The file sepolicy/private/file.te declares nissan_fod_file and assigns it to the standard Android file-type attribute groups: 

# sepolicy/private/file.te

type nissan_fod_file, file_type, data_file_type, core_data_file_type;

file_type

Base Android attribute required for any type used to label files. Enables generic file-system rules.

data_file_type

Marks the type as data storage (like /data subtypes). Required by some framework-level backup/restore macros.

core_data_file_type

Ensures the type inherits from Android's core data group — necessary to satisfy noverify and seinfo checks during system boot validation.

By inheriting these attributes the type slots cleanly into Android's existing policy framework without triggering CTS/VTS compatibility failures, while remaining a completely new namespace that no existing rule can accidentally match.

5. Labelling the Path: file_contexts

Declaring the type is not enough — the kernel must know which filesystem paths should receive the label. sepolicy/private/file_contexts provides the mapping: 

# sepolicy/private/file_contexts

/mnt/product/persist-alliance/fod(/.*)?    u:object_r:nissan_fod_file:s0

Breaking down each field:

/mnt/product/persist-alliance/fod(/.*)?

Regex matching the directory itself and every file inside it recursively. The (/.*)? suffix is standard SELinux regex notation for "this path or anything under it".

u:object_r:nissan_fod_file:s0

Full SELinux security context: user u, role object_r (standard for all file objects), type nissan_fod_file, sensitivity level s0 (MLS not used on this platform).

The label is stamped onto new files at creation time by the kernel's security_fs_use_trans hook. Running restorecon -R /mnt/product/persist-alliance/fod/ applies the label to existing files (needed after OTA deploys the new policy).

6. Granting Access: alliancecarservice_app.te

Finally, the actual permission grant. sepolicy/private/alliancecarservice_app.te adds three allow rules to the car-service domain: 

# sepolicy/private/alliancecarservice_app.te

# 1. Read the persist-alliance directory (needed to list parent paths)
allow alliancecarservice_app { mnt_product_file persist_alliance_file }:dir r_dir_perms;

# 2. Full directory access on the FoD subdirectory (create, list, delete entries)
allow alliancecarservice_app nissan_fod_file:dir { rw_dir_perms create_dir_perms };

# 3. Full file access (open, read, write, create, unlink)
allow alliancecarservice_app nissan_fod_file:file { rw_file_perms create_file_perms };

Macro expansions for reference:

Macro Expands to (key permissions)
r_dir_permsopen read getattr search ioctl
rw_dir_permsr_dir_perms + write add_name remove_name
create_dir_permsrw_dir_perms + create setattr relabelfrom relabelto
rw_file_permsopen read write getattr ioctl lock
create_file_permsrw_file_perms + create setattr unlink rename

Rule 1 grants read-only traversal of the parent directories — required so the process can resolve the full path at runtime. Rules 2 and 3 grant full create/read/write access exclusively to the FoD namespace. No other domain has any of these permissions on nissan_fod_file.

7. The Three Files Together

The complete policy contribution consists of exactly three files in the private policy directory:

sepolicy/private/
├── file.te              ← declares the type
├── file_contexts        ← maps filesystem path → type
└── alliancecarservice_app.te   ← grants permissions to the service domain

Placing them in private/ means the rules are compiled into the system policy partition and cannot be overridden by vendor policy — important for ASPICE traceability and audit.

8. Why This Approach?

 Why Not Linux DAC (chmod/chown)?

DAC (Discretionary Access Control) works by UID/GID ownership. Any process running as the same UID — or as root — bypasses DAC. SELinux MAC is enforced regardless of UID: even root is denied if there is no matching allow rule. Automotive security guidelines require MAC for persistent feature data.

 Why a Dedicated Type (Not Reuse Existing)?

Reusing persist_alliance_file or mnt_product_file would share the label with other services that have legitimate access to those paths. A dedicated nissan_fod_file type enables least-privilege: the blast radius of any vulnerability in another service cannot reach FoD data.

 Why private/ Policy Location?

Android's SELinux build system merges public/, private/, and vendor/ policies separately. Declaring the type in private/ keeps it in the system image and prevents vendor OTA patches from accidentally expanding access to FoD files.

 Why core_data_file_type?

Android CTS/VTS run neverallow checks against core_data_file_type to ensure no dangerous domains (e.g. untrusted_app) can access core data files. By assigning this attribute to nissan_fod_file, the standard platform neverallow rules automatically protect FoD data from all untrusted app domains — zero extra lines needed.

 Why Separate Dir and File Rules?

SELinux distinguishes between :dir and :file object classes. A :file allow alone does not let the process list or create directory entries. Explicit :dir { rw_dir_perms create_dir_perms } ensures the service can initialise the directory on first boot and list entries for file existence checks.

 ASPICE Traceability

Every security requirement in the SRS (Software Requirements Spec) that referred to FoD data protection maps to exactly these three files. SELinux policy is source-controlled and diff-reviewable, satisfying ASPICE SWE.3 (software unit design) and SWE.4 (unit verification) evidence requirements.

9. Verifying the Policy

During development and testing, the policy was verified with:

# 1. Check the label on the filesystem
adb shell ls -Z /mnt/product/persist-alliance/fod/
# Expected: u:object_r:nissan_fod_file:s0

# 2. Confirm process domain
adb shell ps -Z | grep alliancecarservice
# Expected: u:r:alliancecarservice_app:s0

# 3. No AVC denials in logcat for FoD operations
adb logcat | grep "avc: denied" | grep fod
# Expected: (empty — no denials)

# 4. Confirm other domains are blocked — attempt from shell
adb shell ls /mnt/product/persist-alliance/fod/
# Expected: ls: /mnt/product/persist-alliance/fod/: Permission denied

# 5. Static policy audit with sepolicy-analyze
sepolicy-analyze policy.30 neverallow -w

Unit tests in the swut folder verify that FileUtil.checkFileExist() returns the correct result from the service domain. Integration tests (SWIT) ran the full FoD activation flow on a real device and confirmed that fod_data and fod_state files were created with label nissan_fod_file and that no AVC denials appeared in dmesg.
