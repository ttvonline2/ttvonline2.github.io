1. What is FoD and Why It Matters

Feature on Demand (FoD) is a vehicle-software mechanism that lets the OEM (Nissan/Renault) deliver, activate, or reconfigure vehicle software features over-the-air — without a full ECU reflash. Examples include enabling heated seats, unlocking adaptive cruise-control, or changing driving-mode behaviour on demand.

The Android Automotive service (AllianceCarFodService) is the ECU-side broker: it receives execution requests from the OCM (Off-board Communication Manager) — the cloud/OTA layer — validates them, persists parameter changes, and forwards results back to the OCM and to any registered HMI listener.

 OCM (Cloud Side)

Off-board Communication Manager — triggers FoD activation requests, sends FodId + parameters.

 AllianceCarFodService

Android Automotive plug-in service. Validates, persists, and executes feature parameter changes.

 HMI / ECU Listeners

HVAC, Seat, and other services that register as IFodEventListener via AIDL Binder.

2. The Engineering Challenge

A naïve implementation — just writing new parameters directly when the OCM requests it — fails in several real-world scenarios that automotive software must handle:

  • Reboot in the middle of execution — the ECU can lose power at any time. Without persisted state, the FoD process restarts from scratch, potentially leaving the vehicle feature in a half-applied state.
  • Concurrent requests — OCM may send a second activation while the first is still being processed by the downstream HVAC/Seat service. Without a guard, both get applied simultaneously.
  • Stuck execution — if the downstream listener crashes and never calls back, the broker would be locked forever, refusing all further FoD operations.
  • Out-of-order responses — the result callback from listener must be matched to the originating request, not just accepted blindly.

A state machine solves all four problems with a single, provable model.

3. The Seven States

Defined in FodConstant.java, the lifecycle uses numeric IDs that are persisted to disk:

[1] IdleState

Ready to accept any new request. Default state on a clean boot.

[2] ReadyForExecutionState

OCM has sent a "prepare" signal. Service waits for the actual execution payload.

[3] StartExecutionState

Validation in progress. Parameters are being checked against the database.

[4] ExecutionInProgressState

Validated params dispatched to downstream listeners. Awaiting ACK from HVAC/Seat.

[5] FinishExecutionState

Execution result ready. Waits for OCM to call changeState(IDLE) before resetting.

[51] ReadFodApplicabilityState

Reading current feature availability from database for applicability upload.

[52] FinishFodApplicabilityState

Applicability data ready to return to OCM. Awaiting acknowledgement.

Any state that is not [1] Idle rejects new execution requests with ECU_BUSY (8301), ensuring serialised processing. The state IDs intentionally match the numeric codes defined in the ASPICE SRS document so that test engineers could verify state transitions at the protocol level.

4. State Pattern Implementation

The implementation uses the classic GoF State Pattern: an abstract base class FodState declares the contract, and each concrete state encapsulates only the behaviour valid in that state. 

// FodState.java — abstract base (simplified)
public abstract class FodState {

    protected FodClient fodClient;
    protected Handler   fodStateHandler;

    public abstract FodApplicabilityData requestApplicabilityUpload();
    public abstract int                  requestReadState();
    public abstract int                  requestChangeState(int targetState);
    public abstract FodExecutionResultData requestStartExecution(FodExecutionData data);
    public abstract FodParameterData     requestCheckParameter(long fodId);

    // Shared: arm a 20-min watchdog to reset back to Idle
    public void startTimeoutBackToIdle() {
        fodStateHandler.removeCallbacksAndMessages(null);
        fodStateHandler.postDelayed(() -> {
            fodClient.setCurrentState(fodClient.getFodStateInstance(IDLE_STATE));
            saveCurrentStateToDatabase(IDLE_STATE);
        }, DELAY_BACK_TO_IDLE_STATE_TIME_OUT); // 20 mins
    }

    protected void saveCurrentStateToDatabase(int stateId) {
        fodClient.saveCurrentStateToDatabase(stateId);
    }
}

FodClient holds a reference to the current state object and delegates every incoming call to it. Transitioning states is simply swapping that reference: 

// Inside ReadyForExecutionState.requestStartExecution()
// Step 1 — transition to StartExecution
fodClient.setCurrentState(fodClient.getFodStateInstance(START_EXECUTION_STATE));
saveCurrentStateToDatabase(START_EXECUTION_STATE);
startTimeoutBackToIdle();

// Step 2 — validate execution data
FodExecutionResultData result =
    fodClient.getFodExecutionManager().requestCheckExecutionData(executionData);

// Step 3 — if validation OK, notify listeners; else jump to Finish
if (result.getResultCode() == INTERNAL_READY_FOR_UPDATE) {
    fodClient.dispatchedParamChanged(mergedFoDExecutionData);   // → [4]
} else {
    fodClient.setCurrentState(getFodStateInstance(FINISH_EXECUTION_STATE)); // → [5]
    saveCurrentStateToDatabase(FINISH_EXECUTION_STATE);
    startTimeoutBackToIdle();
}

Because every state object knows which transitions are legal and simply ignores or returns ECU_BUSY for all others, no if/else chain is needed in the service layer — the correct behaviour is encoded in the type.

5. State Persistence Across Reboots

The current state integer is written to a plain text file on the persist partition after every transition. The file path is: 

/mnt/product/persist-alliance/fod/fod_state

The write uses FileDescriptor.sync() to force the kernel page cache to disk before the method returns — guaranteeing the value is durable even if power is cut a millisecond later: 

// FodStateDatabaseManager.java
private synchronized void writeDataToFile() {
    FileOutputStream fos = new FileOutputStream(mFileName);
    FileDescriptor   fd  = fos.getFD();

    fos.write(String.valueOf(mCurrentState).getBytes(StandardCharsets.UTF_8));
    fos.write(System.lineSeparator().getBytes(StandardCharsets.UTF_8));
    fos.flush();
    fd.sync();   // ← fsync: data guaranteed on non-volatile storage
}

The FoD parameter database (fod_data) uses the same pattern: every parameter update calls fd.sync() to ensure the approved vehicle-feature configuration survives a reboot.

 fod_state

A single integer: the current state ID (1–52). Read on every boot to resume the pipeline.

 fod_data

One line per FodId: FodId SAFlag SubId-Param_SubId-Param. Synced to disk after every successful execution.

6. Boot-Time Recovery Logic

When FodStateDatabaseManager initialises (singleton, created at service start), it reads the persisted state and applies deterministic recovery rules: 

// FodStateDatabaseManager constructor
private FodStateDatabaseManager() {
    getDataFromFile(mFileName);   // read last state from disk

    if (mCurrentState == 0) {
        // First boot ever — file was empty
        mCurrentState = IDLE_STATE;
        writeDataToFile();
    } else if (mCurrentState == READ_FOD_APPLICABILITY_STATE) {  // 51
        // Rebooted while reading applicability → safe to retry from Idle
        mCurrentState = IDLE_STATE;
        writeDataToFile();
    } else if (mCurrentState == FINISH_FOD_APPLICABILITY_STATE) { // 52
        // Rebooted after applicability was ready but before OCM acked → reset
        mCurrentState = IDLE_STATE;
        writeDataToFile();
    }
    // States 2–5 (execution pipeline) are left as-is
    // so FodClient can decide whether to resume or reset via FodState logic
}

The key insight: applicability states (51, 52) are always safe to reset to Idle because they are read-only queries — re-running them on the next boot has no side effect. Execution states (2–5) are preserved so the higher-level FodClient can decide whether to resume (e.g. re-dispatch to listeners that reconnected) or fail cleanly.

7. 20-Minute Safety Timeout

Every non-Idle state arms a 20-minute Handler.postDelayed() watchdog via startTimeoutBackToIdle(). If the pipeline does not complete (e.g. a downstream service crashes and never ACKs), the watchdog fires and resets to IdleState, persisting state 1 to disk. 

// FodConstant.java
public static final long DELAY_BACK_TO_IDLE_STATE_TIME_OUT = 20 * 60 * 1000L; // 20 minutes

// FodState.java
public void startTimeoutBackToIdle() {
    fodStateHandler.removeCallbacksAndMessages(null); // cancel previous timer
    fodStateHandler.postDelayed(() -> {
        fodClient.setCurrentState(fodClient.getFodStateInstance(IDLE_STATE));
        saveCurrentStateToDatabase(IDLE_STATE);
    }, DELAY_BACK_TO_IDLE_STATE_TIME_OUT);
}

The timer is rearmed on every transition — not just the first — so a legitimate long-running execution (e.g. multiple listener round-trips) gets the full 20 minutes from each step. removeCallbacksAndMessages(null) ensures only one timer is active at any time.

8. Full Execution Flow (Step by Step)

A complete FoD activation from OCM to persistence:

# Caller → Action State Before State After Persisted?
1OCM → requestChangeState(2)[1] Idle[2] ReadyForExecution✅ disk
2OCM → requestStartExecution(data)[2] Ready[3] StartExecution✅ disk
3Internal → FodExecutionManager.checkExecution()[3] Start[3] Start (validating)—
4aValidation OK → dispatch to listeners[3] Start[4] InProgress✅ disk
4bValidation FAIL → skip to Finish[3] Start[5] Finish✅ disk
5Listener ACK → responseChangedSettingParamResult()[4] InProgress[4] InProgress (counting ACKs)—
6All ACKs received → FodStorage.updateFodParam()[4] InProgress[5] Finish✅ disk (params + state)
7OCM → requestChangeState(1)[5] Finish[1] Idle✅ disk

Resumability: If the device reboots between steps 2–6, the persisted state (e.g. 3) is restored on the next boot. The OCM, detecting the stale state via requestReadState(), can decide to cancel (call changeState(1)) or re-attempt the execution — both paths are handled cleanly.

9. Why These Design Choices?

 Why State Pattern (GoF)?

Each state object is independently testable (unit tests cover every state class in the swut folder). Adding a new state requires only a new class — no giant switch in the service. ASPICE SDD documentation maps 1:1 to state classes.

 Why a Plain Text File Instead of SQLite?

A single integer needs a single line. SQLite adds a 300 KB library, schema migrations, and cursor management overhead for no benefit. The plain file is crash-safe with fd.sync() and trivially auditable in logcat.

 Why the Persist Partition?

/mnt/product/persist-alliance/fod/ survives OTA updates and factory-reset (depending on configuration). Vehicle feature entitlements must outlive software updates — a user who paid for heated seats should not lose the feature after a firmware flash.

 Why the 20-Min Timeout?

Automotive standards require services to self-heal. A 20-minute window covers legitimate slow executions (e.g. waiting for vehicle ignition cycle) while ensuring the service is never permanently stuck. The timeout value was agreed with the OCM team based on their worst-case round-trip spec.

 Why Serialize All State Writes?

FodStateDatabaseManager and FodDatabaseManager are singletons with synchronized file-access methods. The service runs callbacks on a dedicated HandlerThread, ensuring all state transitions are single-threaded and the on-disk file is never corrupted by concurrent writes.

 Why Track Individual Listener ACKs?

Multiple AIDL listeners (HVAC + Seat) can be registered for the same FodId. The service issues a unique fodEventId per listener and tracks a HashSet of pending ACKs. Only when all ACKs arrive does it commit the parameter update — preventing partial feature activation.
