1. What is FoD and Why It Matters

Feature on Demand (FoD) is a vehicle-software mechanism that lets the OEM (Nissan/Renault) deliver, activate, or reconfigure vehicle software features over-the-air — without a full ECU reflash. Examples include enabling heated seats, unlocking adaptive cruise-control, or changing driving-mode behaviour on demand.

The Android Automotive service (AllianceCarFodService) is the ECU-side broker: it receives execution requests from the OCM (Off-board Communication Manager) — the cloud/OTA layer — validates them, persists parameter changes, and forwards results back to the OCM and to any registered HMI listener.

 OCM (Plugin Service)

Another AllianceCarFodService plugin service. It communicates directly with the SOMEIP service and provides APIs for other plugin services like FoD.

 AllianceCarFodService

Android Automotive plug-in service. Validates, persists, and executes feature parameter changes.

 HMI / ECU Listeners

HVAC, Seat, and other services that register as IFodEventListener via AIDL Binder.

2. States & Requests Overview

The FoD lifecycle is modelled with 7 states and 5 request types defined in FodConstant.java. Each state ID is a numeric value persisted to disk so the pipeline can survive a reboot.

7 States

[1] IdleState

Ready to accept any new request. Default state on a clean boot.

[2] ReadyForExecutionState

OCM has sent a "prepare" signal. Service waits for the actual execution payload.

[3] StartExecutionState

Validation in progress. Parameters are being checked against the database.

[4] ExecutionInProgressState

Validated params dispatched to downstream listeners. Awaiting ACK from HVAC/Seat.

[5] FinishExecutionState

Execution result ready. Waits for OCM to call changeState(IDLE) before resetting.

[51] ReadFodApplicabilityState

Reading current feature availability from the database for applicability upload.

[52] FinishFodApplicabilityState

Applicability data ready to return to OCM. Awaiting acknowledgement.

5 Request Types

requestChangeState()

Ask the service to move to a target state (e.g. Idle → ReadyForExecution).

requestReadState()

Query the current state without changing it. Used by OCM to detect stale or in-progress sessions.

requestStartExecution()

Send a FoD execution payload (FodId + parameters) to be validated and applied.

requestCheckParameter()

Check the stored parameters for a given FodId.

requestApplicabilityUpload()

Trigger a read of current feature-availability data and return it to OCM.

 My Contribution — Scenario Coverage

During the design phase, the team had the 7 states and 5 requests defined, but the full transition matrix had not been worked out. I initiated a discussion with the OEM (Nissan/Renault) to map out every combination — any of the 5 requests arriving in any of the 7 states. The goal was to make sure we covered all 35 scenarios, not just the "happy path". This work resulted in the complete state-transition table that became part of the ASPICE SDD and drove the unit-test cases for every state class.

3. Why State Pattern Design Choice?

 Behaviour Is Encoded in the Type

Each state class encapsulates only the behaviour that is valid in that state. There is no giant switch or if/else chain in the service layer. An illegal request in the wrong state simply returns ECU_BUSY (8301) — enforced by the state object itself.

 Each State Is Independently Testable

Unit tests (in the swut folder) cover every concrete state class in isolation. You can verify all 35 request-in-state scenarios without spinning up the full service.

 Easy to Extend

Adding a new state only requires a new class that extends FodState. No existing code needs to change. This matches the ASPICE requirement that SDD changes are traceable and isolated.

 1:1 Mapping to ASPICE SDD

Each state class maps directly to a state defined in the ASPICE Software Design Document. Test engineers can verify state transitions at the protocol level because the state IDs in code match the numeric codes in the specification.

 Reboot Safety via Persistence

Because the current state is a single integer, it can be written to disk after every transition using FileDescriptor.sync(). The state machine resumes correctly on the next boot without any complex recovery logic.

 Self-Healing via Timeout

Every non-Idle state arms a 20-minute Handler.postDelayed() watchdog. If a downstream service crashes and never ACKs, the watchdog resets the machine to Idle — the service never gets permanently stuck.
