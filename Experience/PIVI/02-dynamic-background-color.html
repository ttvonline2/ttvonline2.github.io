1. The Feature — What It Looks Like

On the Spotify now-playing screen, the background is not a flat dark colour. Instead, it displays a soft gradient that mirrors the dominant hue of the current track's album art:

  • A warm red album (e.g. Taylor Swift) → the background glows with a muted crimson gradient.
  • A cool blue album (e.g. Coldplay) → the background shifts to a calm teal/navy tone.
  • When the user skips a track, the background smoothly animates (400 ms ColorAnimation) from the old colour to the new one.
  • In dark mode the fallback colour is #21252E; in light mode it is #D2D2D2 — both neutral defaults used before the first artwork is loaded.

The effect is the same UX pattern popularised by Spotify's desktop and mobile apps, adapted for an embedded QNX/Qt automotive HMI with no access to the Android Palette API.

2. The Challenge: No Palette Library on QNX

Android's androidx.palette library and iOS's UIColor+DominantColor are not available on our QNX + Qt stack. The only tooling available was:

Available

QImage::pixelColor(x, y) — pixel-level read of any in-memory image.
QColor::lightness() and QColor::saturation() — HSL components.
QML ColorAnimation — animated property transitions.
QML Image::grabToImage() — render QML image to a QImage.

Not Available

Android Palette API. iOS UIKit color extraction. Python/ML color clustering. Any third-party native library (build system restrictions). GPU compute shaders for this use case.

The solution was a custom pixel-scan heuristic written in C++ that finds the pixel whose HSL values are closest to a target "vibrant" profile — which matches the perceptual quality of the Spotify-style dynamic background without k-means clustering or any ML.

3. The Vibrant Color Algorithm

The algorithm answers: "Which pixel in this image is the most vibrant?" by defining vibrant as the closest to a target lightness and saturation in HSL colour space. 

// HmiOnlineMediaInterface.cpp — tuneable constants
#define OFFSET              20    // Tolerance band (unused in final version)
#define JUMP_RANGE          1     // Scan every pixel (1 = full resolution)
#define TARGET_LIGHTNESS   155    // Vibrant: medium-light (0–255 scale)
#define TARGET_SATURATION  129    // Vibrant: moderately saturated (0–255 scale)
void HmiOnlineMediaInterface::requestProcessVibrantColor()
{
    QString local_path = "/var/onlinemedia/hmi/albumArt.png";
    QImage img(local_path);

    if (img.width() != 0) {
        int    max_diff = 510;   // worst possible diff = |255-0| + |255-0| = 510
        QColor result;

        for (int w = 0; w < img.width(); w += JUMP_RANGE) {
            for (int h = 0; h < img.height(); h += JUMP_RANGE) {
                QColor cl = img.pixelColor(w, h);

                // "distance" from the ideal vibrant colour in HSL space
                int diffValue = abs(cl.lightness()   - TARGET_LIGHTNESS)
                              + abs(cl.saturation()  - TARGET_SATURATION);

                if (diffValue <= max_diff) {
                    result   = cl;
                    max_diff = diffValue;   // progressively tighten the winner
                }
            }
        }
        emit sigUpdateVibrantColor(result.name());   // emit hex string e.g. "#a8312e"
    }
}

How it works visually:

Step 1 — HSL Distance

For every pixel, compute |lightness − 155| + |saturation − 129|. This is an L1 (Manhattan) distance in a 2D HSL subspace. Low distance = colour close to the target vibrant profile.

Step 2 — Progressive Winner

max_diff starts at 510 (maximum possible) and is updated each time a closer pixel is found. After scanning all pixels, result is the global nearest-vibrant pixel — an O(W × H) linear scan.

Step 3 — Why HSL Not RGB?

HSL separates colour identity (hue) from brightness (lightness) and purity (saturation). An RGB distance would treat dark-red and bright-red as completely different. HSL lets us independently target "medium brightness, medium saturation" — the sweet spot for readable background gradients.

Step 4 — Fixed Targets (155, 129)

These values were tuned empirically by running the algorithm against 50+ album artworks and evaluating visual harmony on the head unit display. They select colours that are neither too dark (invisible on dark UI) nor too bright (overwhelming on the screen).

The input image is scaled to 100×100 pixels before the scan (see Step 1), making the worst-case loop 100 × 100 = 10,000 iterations — completing in < 1 ms on the head unit CPU.

5. Step 1 — QML Saves Artwork, Triggers C++

When the album art image finishes loading in QML (Image.Ready), the QML code grabs a 100×100 thumbnail to disk and then calls back into C++: 

// PlayerSpotifyView.qml — inside the Image element's onStatusChanged handler
if (status === Image.Ready) {
    albumArt.grabToImage(function(result) {
        var saved = result.saveToFile("/var/onlinemedia/hmi/albumArt.png");
        console.log("PlayerSpotifyView saved " + saved);
        iMediaInterface.requestProcessVibrantColor();  // invoke C++ method
    }, Qt.size(100, 100));   // ← scale down to 100×100 before saving
}

grabToImage(callback, Qt.size(100,100))

Renders the displayed QML Image element (already decoded, cached locally) into an offscreen surface and converts it to a QImage at 100×100. The downscale reduces pixel-scan iterations from ~90,000 (300×300) to 10,000.

saveToFile(path)

Writes the 100×100 PNG to /var/onlinemedia/hmi/albumArt.png — the same path the disk cache uses. The C++ side reads this exact file. Using a fixed path (not a temp file) avoids filename-passing complexity in the signal: the C++ function needs no argument.

6. Step 2 — C++ Scans Pixels, Emits Color

requestProcessVibrantColor() is declared as Q_INVOKABLE on HmiOnlineMediaInterface, making it callable directly from QML via the registered context property iMediaInterface: 

// HmiOnlineMediaInterface.h
Q_INVOKABLE void requestProcessVibrantColor();

// Signal emitted when the best colour is found
void sigUpdateVibrantColor(QString vibrantColor);

After the pixel scan the best colour's hex string (e.g. "#a8312e") is emitted via sigUpdateVibrantColor. Qt's signal/slot mechanism queues this emission on the main thread's event loop, ensuring the QML update happens on the UI thread regardless of where the scan was initiated.

7. Step 3 — Signal Chain Through the Adapter

The colour signal travels from the interface layer through the adapter to reach the QML root object: 

// HmiOnlineMediaAppMain.cpp — wiring at startup
connect(m_onlineMediaInterface,       SIGNAL(sigUpdateVibrantColor(QString)),
        m_hmiPlayerSpotifyViewAdapter, SLOT(onUpdateVibrantColor(QString)));
// PlayerSpotifyViewAdapter.cpp
void PlayerSpotifyViewAdapter::onUpdateVibrantColor(QString vibrantColor) {
    MTLOG_I(TAG, "onUpdateVibrantColor: %s", vibrantColor.toStdString().c_str());
    emit sigOnUpdateVibrantColortoQML(vibrantColor);   // forward to QML root
}
// PlayerSpotifyViewAdapter::connectBetweenUIandAdapter()
connect(this, SIGNAL(sigOnUpdateVibrantColortoQML(QString)),
        m_rootObject, SIGNAL(sigOnUpdateVibrantColor(QString)));

The adapter layer is the same Adapter pattern used for all player events in the codebase: it bridges the service-layer C++ signals to QML root-object signals, keeping the interface layer unaware of QML and the QML side unaware of the C++ service layer.

8. Step 4 — QML Animates the Gradient

The QML player view maintains two colour properties — vibrantColor (currently rendered) and new_vibrantColor (incoming target) — and uses a ColorAnimation plus a Timer to animate between them: 

// PlayerSpotifyView.qml — properties
property string vibrantColor:     objectStyler.timeMode === "dark" ? "#21252E" : "#D2D2D2"
property string new_vibrantColor: objectStyler.timeMode === "dark" ? "#21252E" : "#D2D2D2"
// Signal handler — called when C++ emits the new colour
onSigOnUpdateVibrantColor: {
    if (vibColor !== new_vibrantColor) {
        new_vibrantColor = vibColor      // 1. store the new target
        bg_animation.stop()
        bg_animation.start()             // 2. start gradient colour animation
        tm_delayVibrant.stop()
        tm_delayVibrant.start()          // 3. start 400ms timer
    }
}
// 400ms delay timer — commits the new colour to the live property
Timer {
    id: tm_delayVibrant
    interval: 400; running: false; repeat: false
    onTriggered: vibrantColor = new_vibrantColor   // swap after animation completes
}
// Background gradient rectangle using the animated colour
Rectangle {
    anchors.fill: parent
    id: bg_spotify
    gradient: Gradient {
        GradientStop {
            position: 0.0
            color: vibrantColor

            ColorAnimation on color {
                id: bg_animation
                to: new_vibrantColor
                duration: 400              // 400ms smooth transition
            }
        }
        GradientStop { position: 1.0; color: "transparent" }
    }
    opacity: 0.3   // subtle overlay — does not overpower album art
}

GradientStop at 0.0

Top of the background rectangle gets the vibrant colour (animated). Creates a glow emanating from the top of the screen.

GradientStop at 1.0 — transparent

Bottom fades to transparent, so album art and controls are not obscured by the colour fill.

opacity: 0.3

The tinted layer is semi-transparent — the dark background base rectangle shows through, maintaining readability of white text.

Two-property pattern (vibrantColor / new_vibrantColor)

Separating "current" from "target" lets the animation play fully without jitter if a second track change arrives mid-animation — the timer is restarted and only the latest colour is applied.

9. Why These Design Choices?

 Why Pixel Scanning Instead of k-Means Clustering?

k-Means requires either a native C++ library (unavailable under build constraints) or a significant implementation effort. The pixel-scan heuristic achieves a perceptually equivalent result — "find the most vibrant pixel" — in < 1 ms using only Qt's QColor API. For a background gradient that is rendered at 30% opacity, the difference from a full palette extraction is invisible to the driver.

 Why Downscale to 100×100 Before Scanning?

A Spotify artwork is 300×300 to 640×640 px. Scanning at full resolution would be 90,000–409,600 iterations, potentially causing a UI frame drop on the single-core QNX HMI process. Downscaling to 100×100 caps the loop at 10,000 iterations while preserving the overall colour distribution — album art colour is globally distributed, not pixel-precise.

 Why Save to a Fixed File Path?

requestProcessVibrantColor() takes no argument. The fixed path /var/onlinemedia/hmi/albumArt.png acts as a contract between QML and C++. This avoids marshalling a path string across the QML/C++ boundary and eliminates the need for a dedicated IPC message just to pass a filename.

 Why HSL and Not RGB Distance?

RGB Euclidean distance would find the pixel most "similar to a reference RGB colour." That requires choosing a reference colour, which varies per artwork. HSL lightness and saturation targets define an abstract vibe — medium brightness, medium purity — that reliably picks a pleasing accent colour from any image, regardless of its dominant hue.

 Why 400 ms Animation Duration?

400 ms is long enough to be perceived as a smooth, intentional transition (not a jarring snap) but short enough not to still be animating when the user has already navigated away. It matches the default Spotify app's background transition duration — the same number passengers would have seen on their phones.

 Why opacity: 0.3?

The PIVI design guidelines required that the album art remain the visually dominant element. A full-opacity coloured background would overpower the artwork and make white text labels harder to read at certain colour values. 30% opacity creates the ambient glow effect without competitive interference.
