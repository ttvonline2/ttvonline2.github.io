1. The Problem — Why Caching Was Needed

The PIVI Online Media application surfaces content from Spotify, TuneIn, Deezer, Kuwo, Alexa, and Tmall Genie. Every browse list item, search result, and now-playing screen loads album artwork images from external CDNs over HTTPS.

On the automotive head unit the network path is a 4G/LTE modem — latency spikes are common. Without caching:

  • Every navigation to a browse list re-fetched artwork that was just visible seconds ago.
  • Scrolling back and forth in a playlist triggered tens of duplicate HTTP requests per second.
  • Switching providers (e.g. Spotify → TuneIn → Spotify) re-downloaded the same cover images.
  • On packet loss or signal drop, images stayed blank; there was no local fallback.

The fix: teach Qt's QML engine to serve all image URLs from a local disk cache first, only going to the network on a cache miss.

 Before Caching

Each image load: DNS → TLS handshake → HTTP GET → decode. Typical: 300–800 ms per image.

 After Caching

Cache hit: direct file read from /var/onlinemedia/hmi/. Typical: < 5 ms per image.

2. Qt's Network Stack in QML

In Qt Quick (QML), when an Image { source: "https://…" } element is rendered, the QML engine internally creates a QNetworkRequest and dispatches it through a QNetworkAccessManager (NAM).

By default each QQmlEngine creates its own NAM with no cache attached. Qt provides a factory hook to replace that default: 

// Inject a custom factory before loading any QML
engine->setNetworkAccessManagerFactory(myFactory);
// From that point on, every QML Image/XmlHttpRequest goes through myFactory.create()

The factory pattern is ideal here because each QML engine owns its own NAM, and the factory is called once per engine construction — meaning the cache directory and size are configured in exactly one place and shared consistently across all views.

QQmlNetworkAccessManagerFactory

Abstract factory — create(QObject* parent) must return a configured NAM. Called once per QML engine.

QNetworkAccessManager

The central HTTP dispatcher. Attach a QAbstractNetworkCache to it and it automatically serves GET responses from cache.

QNetworkDiskCache

Qt's built-in disk cache. Stores full HTTP responses (headers + body) keyed by URL. Respects Cache-Control headers.

3. Solution Architecture

Three classes collaborate to deliver the caching layer:

QQmlNetworkAccessManagerFactory
          │
          └── DiskCacheFactory              ← our factory (configures cache path + size)
                    │
                    └── CachingNetworkAccessManager   ← our NAM (inherits QNetworkAccessManager)
                              │
                              ├── QNetworkDiskCache   ← Qt disk cache (10 MB, /var/onlinemedia/hmi/)
                              └── SSL error bypass    ← ignoreSSLErrors slot

The QML engine calls DiskCacheFactory::create() once. The returned CachingNetworkAccessManager is then used for the entire lifetime of that engine. Every Image { source: "https://…" } goes through this NAM transparently.

4. DiskCacheFactory — The Factory Hook

// DiskCacheFactory.h
class DiskCacheFactory : public QQmlNetworkAccessManagerFactory {
public:
    DiskCacheFactory(int cacheSize);
    QNetworkAccessManager* create(QObject *parent) override;
private:
    int mCacheSize;
    void validateDownloadFilePath(std::string path);
};
// DiskCacheFactory.cpp
#define HMI_ONLINEMEDIA_PATH "/var/onlinemedia/hmi"

QNetworkAccessManager* DiskCacheFactory::create(QObject *parent) {
    QNetworkAccessManager *nam = new CachingNetworkAccessManager(parent);
    QNetworkDiskCache *diskCache = new QNetworkDiskCache(nam);

    QString cacheFolder = QString(HMI_ONLINEMEDIA_PATH);
    validateDownloadFilePath(HMI_ONLINEMEDIA_PATH);         // create dir if missing

    diskCache->setCacheDirectory(cacheFolder);
    diskCache->setMaximumCacheSize(mCacheSize);              // 10 MB

    nam->setCache(diskCache);
    MTLOG_I(TAG, "Installing network cache Folder size: %d KB, Path: %s",
            mCacheSize / 1024, cacheFolder.toStdString().c_str());
    return nam;
}

void DiskCacheFactory::validateDownloadFilePath(std::string path) {
    path += "/";
    if (access(path.c_str(), F_OK) != F_OK) {
        if (access("/var/onlinemedia/", F_OK) != F_OK) {
            mkdir("/var/onlinemedia/", S_IRWXU | S_IRWXG | S_IRWXO);
        }
        mkdir(path.c_str(), S_IRWXU | S_IRWXG | S_IRWXO);
    }
}

validateDownloadFilePath() ensures the directory exists before the cache tries to write to it — critical on first boot when the persist partition may be completely empty. It uses POSIX access() + mkdir() rather than Qt helpers to avoid a Qt dependency in what is essentially a filesystem bootstrap step.

5. CachingNetworkAccessManager — SSL + Cache

// CachingNetworkAccessManager.h
class CachingNetworkAccessManager : public QNetworkAccessManager {
    Q_OBJECT
public:
    CachingNetworkAccessManager(QObject *parent = 0);
public slots:
    void ignoreSSLErrors(QNetworkReply* reply, QList<QSslError> errors);
};
// CachingNetworkAccessManager.cpp
CachingNetworkAccessManager::CachingNetworkAccessManager(QObject *parent)
    : QNetworkAccessManager(parent)
{
    QObject::connect(
        this, SIGNAL(sslErrors(QNetworkReply*, QList<QSslError>)),
        this, SLOT(ignoreSSLErrors(QNetworkReply*, QList<QSslError>))
    );
}

void CachingNetworkAccessManager::ignoreSSLErrors(QNetworkReply *reply,
                                                   QList<QSslError> errors)
{
    reply->ignoreSslErrors(errors);
}

Inheriting QNetworkAccessManager and attaching the cache via nam->setCache(diskCache) is enough to activate caching — Qt's base class intercepts every createRequest() call, checks the cache for a fresh response, and returns a synthetic reply from disk if found. No override of createRequest() is needed.

The ignoreSSLErrors slot is deliberate: content CDNs used by Spotify and Deezer sometimes serve artwork over subdomains with certificate chains not present in the QNX root CA bundle. Ignoring SSL errors only for image resources (which are public CDN assets, not authentication endpoints) was an accepted pragmatic decision reviewed with the security team.

6. Wiring into All QML Engine Views

The application hosts four independent QML engines (main view, speech mode, Alexa mode, Tmall mode). Each gets its own DiskCacheFactory pointing to the same directory — so they all share the same on-disk cache pool: 

// HmiOnlineMediaAppMain.cpp — initialising each view

// Main online-media view (Spotify, TuneIn, Deezer, Kuwo…)
m_view->engine()->setNetworkAccessManagerFactory(new DiskCacheFactory(1024 * 1024 * 10));

// Speech mode overlay
m_viewSpeechMode->engine()->setNetworkAccessManagerFactory(new DiskCacheFactory(1024 * 1024 * 10));

// Alexa mode overlay
m_viewAlexaMode->engine()->setNetworkAccessManagerFactory(new DiskCacheFactory(1024 * 1024 * 10));

// Tmall Genie overlay
m_viewTmallMode->engine()->setNetworkAccessManagerFactory(new DiskCacheFactory(1024 * 1024 * 10));

Cache size: 10 MB (1024 × 1024 × 10 bytes) per factory instance. Because all four factories write to the same directory and QNetworkDiskCache uses URL-keyed files, cached responses are reused across all four engines automatically — an album-art image fetched by the main view is immediately available to the Alexa overlay with no additional network request.

7. On-Device Storage Path

/var/onlinemedia/hmi/

/var on the QNX target is a writable runtime partition that persists across application restarts but is cleared on a full factory reset. This makes it appropriate for a media image cache: content is warm after the first session but does not accumulate indefinitely.

 Cache Format

Qt's QNetworkDiskCache stores each response as two files: a binary header file (metadata, expiry, ETags) and a body file. Directory entries are keyed by a hash of the URL.

 Cache Invalidation

Qt honours Cache-Control: max-age and ETag headers from the CDN. When the TTL expires, the next request performs a conditional GET (If-None-Match) and refreshes only stale entries.

 Eviction

setMaximumCacheSize(10 MB) causes QNetworkDiskCache to evict LRU entries when the limit is reached. Album art for recently played tracks is always hot.

8. Why These Design Choices?

 Why QNetworkDiskCache Instead of a Custom Cache?

Qt's built-in disk cache correctly handles HTTP semantics: ETags, Cache-Control, Vary headers, conditional GETs. A hand-rolled URL → file map would skip these, causing stale artwork on CDN image rotation. The built-in solution gives HTTP-correct behaviour for free.

 Why the Factory Pattern?

QQmlNetworkAccessManagerFactory is Qt's official injection point. It is called once per engine, its return value is owned by the engine, and it requires zero QML changes — all four view files work without modification. Any alternative (monkey-patching, QML XMLHttpRequest wrappers) would require modifying every image source binding.

 Why 10 MB?

Spotify/TuneIn artwork is typically 300×300 JPEG ≈ 15–40 KB. 10 MB holds roughly 250–650 cached images — more than enough to cover a full browse session's thumbnails. The QNX target had limited flash; 10 MB was the agreed ceiling with the system integrator.

 Why /var/onlinemedia/hmi/?

/var is writable from the HMI process without elevated privileges. The persist partition is reserved for settings that survive factory reset. A media thumbnail cache does not need to survive factory reset; /var is the correct tier.

 Why Share the Same Directory Across All 4 Engines?

Qt's disk cache uses URL-hash-keyed filenames. Multiple NAM instances pointing to the same directory share the same cache pool without conflicts — a hit in any engine benefits all. Separate directories would balloon storage usage and eliminate cross-engine warm-up.

 Why Ignore SSL Errors?

The QNX root CA store on the head unit was not updated to include newer CDN intermediate CAs (Spotify's, Deezer's). The images are unauthenticated public CDN assets — there is no security risk in skipping certificate validation for them specifically. The alternative was maintaining a custom CA bundle, which would require OTA updates whenever CDN providers rotated certificates.
