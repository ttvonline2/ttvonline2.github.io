# Brief
- **What is this:** A behavioral design pattern that allows an object to **alter its behavior when its internal state changes**, by delegating state-specific logic to separate state objects rather than piling up `if/else` or `switch` chains.
- **Ultimate Goal:** Model entities that have distinct modes (e.g. a media player: playing / paused / stopped) in a way that is easy to extend with new states without touching existing code.

---

# Details

## Context
> *"The object whose behavior varies with state — it delegates all state-specific work to the current State object."*

- Explain: Holds a reference to the current `State` object and forwards requests to it. The rest of the system only talks to the Context, never to individual states directly
- Goals: Centralize state transitions; keep callers decoupled from which state is active

## State Interface
> *"Defines the contract that every concrete state must fulfill."*

- Explain: An interface (or abstract class) declaring the actions the Context can trigger (e.g. `play()`, `pause()`, `stop()`). Each concrete state implements this interface
- Goals: Enforce a uniform API across all states so the Context can call them interchangeably
- Example: TODO

---

## Concrete States
> *"Each class encapsulates the behavior for one specific state."*

- Explain: Implements the State interface for a particular mode. Handles the action *and* decides if/when to tell the Context to switch to another state
- Goals: Isolate state-specific logic — adding a new state means a new class, not editing every `if/else` branch
- Example: TODO (FOD service — implemented State pattern to manage fingerprint sensor modes cleanly without growing a switch tree)

---

## State Transitions
> *"The rules for moving from one state to another live inside the states themselves (or the Context)."*

- Explain: A state method can call `context.setState(new OtherState())` when conditions are met. Transitions are explicit and traceable rather than scattered `flag = true` assignments
- Goals: Make the lifecycle of an object readable — you can draw a state diagram directly from the code

## Quick Memory Hook

```
Context        → holds current state, delegates calls to it
State interface → common API all states implement
Concrete State  → one class per mode, owns its own logic
Transition      → state (or context) swaps to the next state
Benefit         → no giant switch; new state = new class only
```
