# Brief
- **What is this:** A creational design pattern that ensures a class has **only one instance** for the lifetime of the application, and provides a global access point to it.
- **Ultimate Goal:** Control access to a shared, expensive, or stateful resource (DB connection, config, logger) so that the whole system shares one consistent instance instead of creating duplicates.

---

# Details

## Single Instance Guarantee
> *"Ensure a class has only one instance and provide a global point of access to it."*

- Explain: The constructor is private; the class itself controls creation. A static reference holds the one instance and reuses it on every subsequent call
- Goals: Prevent duplicate instances that could cause inconsistent state or wasted resources

## Lazy vs Eager Initialization
> *"Decide whether the instance is created at startup or only when first needed."*

- Explain:
  - **Eager** — instance created when the class loads; simple but wastes memory if never used
  - **Lazy** — instance created on first `getInstance()` call; efficient but needs thread-safety guards
- Goals: Balance startup cost against memory footprint

## Thread Safety
> *"In a multi-threaded environment, two threads must not create two separate instances simultaneously."*

- Explain: Use `synchronized`, double-checked locking, or an inner static holder class (initialization-on-demand) to ensure only one instance is ever created even under concurrency
- Goals: Prevent race conditions that silently break the singleton guarantee

## When to Use vs Avoid
> *"Singleton is powerful but overuse turns it into a global variable anti-pattern."*

- Explain:
  - **Use** when exactly one shared resource makes sense: config manager, thread pool, analytics logger
  - **Avoid** when it hides dependencies, makes testing hard (can't inject a mock), or couples unrelated modules
- Goals: Apply intentionally; prefer dependency injection to pass the singleton where needed rather than reaching for it globally

## Quick Memory Hook
