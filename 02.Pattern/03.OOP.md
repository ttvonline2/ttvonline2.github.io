# Brief
- **What is this:** A programming paradigm that organizes code around **objects** — bundles of data (fields) and behavior (methods) — built on four core pillars: Encapsulation, Abstraction, Inheritance, and Polymorphism.
- **Ultimate Goal:** Write code that mirrors real-world concepts, is easy to extend without breaking existing parts, and scales cleanly as systems grow in complexity.

---

# Details

## Encapsulation
> *"Bundle data and the methods that operate on it together, and restrict direct access to internal state."*

- Explain: Hide internal details behind a public interface. Fields are `private`; controlled access is provided via getters/setters or methods
- Goals: Prevent external code from putting an object into an invalid state; changes to internals don't ripple outward

## Abstraction
> *"Show only what is necessary; hide the complexity underneath."*

- Explain: Expose a simplified interface (abstract class, interface) while the concrete implementation stays hidden from callers
- Goals: Let callers depend on *what* something does, not *how* — making swapping implementations trivial

## Inheritance
> *"A class can derive behavior and structure from a parent class."*

- Explain: Child class reuses and extends parent class logic. Establishes an **is-a** relationship
- Goals: Avoid duplicating shared logic; establish clear hierarchies

## Polymorphism
> *"The same interface can refer to objects of different underlying types."*

- Explain: A single method call behaves differently depending on the actual runtime object (method overriding / interface implementation)
- Goals: Write generic code that works across many types without `instanceof` chains

## Quick Memory Hook

```
Encapsulation → hide internal state, expose clean API
Abstraction   → show what, hide how
Inheritance   → child reuses and extends parent (is-a)
Polymorphism  → same call, different behavior at runtime
```
