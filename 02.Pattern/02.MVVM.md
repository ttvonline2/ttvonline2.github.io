# Brief
- **What is this:** A structural design pattern that separates UI (View) from business logic (ViewModel) and data (Model), enabling clean two-way data binding and testable UI logic.
- **Ultimate Goal:** Build UIs where the View is dumb (just renders), the ViewModel is independently testable, and the Model stays pure — so the whole thing scales and stays maintainable as complexity grows.

---

# Details

## Model
> *"Represents the data and business logic of the application."*

- Explain: Pure data layer — data classes, repositories, domain logic. Has **no knowledge** of the UI whatsoever
- Goals: Keep business rules isolated so they can be tested without spinning up any UI

## View
> *"Displays data and forwards user interactions to the ViewModel."*

- Explain: The UI layer (Activity, Fragment, XML layout, Composable). It **observes** ViewModel state and reacts — it never holds logic or makes decisions
- Goals: Keep the View as thin as possible so UI changes don't ripple into logic

---

## ViewModel
> *"Exposes streams of data relevant to the View and handles user interaction logic."*

- Explain: The bridge between Model and View. Holds UI state, processes user events, calls repositories — but never references a View directly
- Goals: Survive configuration changes (rotation), be testable in pure JUnit without Android dependencies

## Data Binding / Observation
> *"The View automatically reflects ViewModel state changes without manual wiring."*

- Explain: View subscribes to `LiveData`, `StateFlow`, or a reactive stream. When ViewModel pushes a new state, the View updates itself
- Goals: Eliminate boilerplate `setText`/`setVisibility` calls; reduce the chance of stale UI

## Quick Memory Hook

```
Model  → data + rules, knows nothing about UI
View   → renders state, forwards events, zero logic
ViewModel → holds state, handles events, calls Model
Binding → View reacts automatically to ViewModel state
```
